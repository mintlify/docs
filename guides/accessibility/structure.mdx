Well-structured content is easier to navigate and understand, especially for screen reader users who rely on headings to move through pages and people who use keyboard navigation.

## Use proper heading hierarchy

Each page should have a single H1 heading, which is defined by the `title:` property in a page's frontmatter. Use additional headings in order without skipping. For example, don't skip from H2 to H4.

```mdx
<!-- Good -->
# Page title (H1)

## Main section (H2)

### Subsection (H3)

### Another subsection (H3)

## Another main section (H2)

<!-- Bad -->
# Page title (H1)

## Main section (H2)

#### Subsection (H4)

### Another subsection (H3)
```

Headings at the same level should have unique names.

```mdx
<!-- Good -->
## Accessibility tips (H2)

### Write effective alt text (H3)

### Use proper color contrast (H3)

<!-- Bad -->
## Accessibility tips (H2)

### Tip (H3)

### Tip (H3)
```

### Why heading hierarchy matters

Screen readers use headings to build a page outline that users can navigate. When you skip heading levels or use headings out of order, you create a confusing navigation experience. Users may miss important sections or struggle to understand how content is organized.

Proper heading hierarchy also helps with:
- SEO and search engine understanding
- Keyboard navigation efficiency
- Content scanning and comprehension
- Automated table of contents generation

## Write descriptive link text

Link text should be meaningful and connected to the destination. Avoid vague phrases like "click here" or "read more."

```mdx
<!-- Good -->
Learn how to [configure your navigation](/organize/navigation).

<!-- Unclear relation between link text and destination -->
[Learn more](/organize/navigation).
```

### Best practices for link text

**Be specific about the destination**

```mdx
<!-- Good -->
Read the [WCAG 2.2 guidelines](https://www.w3.org/WAI/WCAG22/quickref/) for detailed accessibility standards.

<!-- Too vague -->
[Click here](https://www.w3.org/WAI/WCAG22/quickref/) for accessibility standards.
```

**Make links understandable out of context**

Screen reader users often navigate by jumping between links. Each link should make sense on its own:

```mdx
<!-- Good -->
- [Install the Mintlify CLI](/installation)
- [Configure your navigation](/organize/navigation)
- [Deploy your documentation](/deploy/deployments)

<!-- Not helpful when read in isolation -->
- [Get started](/installation)
- [Learn more](/organize/navigation)
- [Read the guide](/deploy/deployments)
```

**Avoid URL text for links**

```mdx
<!-- Good -->
Visit the [Mintlify dashboard](https://dashboard.mintlify.com) to manage your documentation.

<!-- Less accessible -->
Visit https://dashboard.mintlify.com to manage your documentation.
```

If you must show a URL, make it the link text:

```mdx
Visit [https://dashboard.mintlify.com](https://dashboard.mintlify.com) to manage your documentation.
```

**Keep link text concise**

Long link text is harder to understand and interact with:

```mdx
<!-- Good -->
Learn about [authentication setup](/deploy/authentication-setup).

<!-- Too long -->
[Learn about how to set up authentication for your documentation site including SSO and user groups](/deploy/authentication-setup).
```

## Keep content scannable

Break up long paragraphs and use formatting to help users find information quickly.

### Use lists for steps and options

Lists make sequential information and options easier to scan:

```mdx
<!-- Good -->
To deploy your documentation:

1. Commit your changes to Git
2. Push to your repository
3. Wait for the build to complete
4. Verify your changes in production

<!-- Harder to scan -->
To deploy your documentation, commit your changes to Git, then push to your repository, wait for the build to complete, and verify your changes in production.
```

### Break up long paragraphs

Aim for 3-4 sentences per paragraph. Long blocks of text are difficult to read and understand:

```mdx
<!-- Good -->
Mintlify automatically generates an MCP server for your documentation. The server is hosted at the /mcp path of your documentation URL.

For public documentation, your MCP server is available to anyone. It searches all indexed public pages.

<!-- Harder to read -->
Mintlify automatically generates an MCP server for your documentation and the server is hosted at the /mcp path of your documentation URL. For public documentation, your MCP server is available to anyone and it searches all indexed public pages.
```

### Highlight important information

Use callouts to draw attention to critical information:

```mdx
<Warning>
Breaking changes in version 2.0 require updates to your configuration file.
</Warning>
```

## Use proper table structure

Use tables sparingly and only for tabular data that has meaning inherited from the column and row headers.

When using tables, include headers so screen readers can associate data with the correct column:

```mdx
| Feature | Status | Description |
| ------- | ------ | ----------- |
| Search  | Active | Full-text search across documentation |
| Analytics | Active | Track page views and user behavior |
| Comments | Beta | Collaborative feedback on pages |
```

### When not to use tables

Avoid using tables for layout purposes. Use proper HTML structure or Mintlify components instead:

```mdx
<!-- Don't use tables for layout -->
| ![Icon](/icon.png) | Feature description goes here |

<!-- Use components instead -->
<Card title="Feature name" icon="icon-name">
  Feature description goes here
</Card>
```

### Complex tables

For complex tables with multiple header levels or merged cells, consider:
- Breaking the table into multiple simpler tables
- Providing a text summary before the table
- Using a different format like a list or definition list

## Use semantic HTML elements

When you need to use HTML in your MDX files, use semantic elements that convey meaning:

```html
<!-- Good -->
<nav>
  <ul>
    <li><a href="/guide">Guide</a></li>
    <li><a href="/api">API</a></li>
  </ul>
</nav>

<!-- Less semantic -->
<div>
  <div>
    <div><a href="/guide">Guide</a></div>
    <div><a href="/api">API</a></div>
  </div>
</div>
```

### Common semantic elements

- `<nav>` for navigation sections
- `<article>` for self-contained content
- `<section>` for thematic grouping
- `<aside>` for tangentially related content
- `<header>` for introductory content
- `<footer>` for closing content
- `<main>` for primary page content

## Provide context for abbreviations

Define abbreviations and acronyms when first used:

```mdx
<!-- Good -->
The Web Content Accessibility Guidelines (WCAG) provide standards for accessible web content.

<!-- Unclear -->
Follow WCAG standards for your documentation.
```

For subsequent uses, you can use the abbreviation alone. Consider using the HTML `<abbr>` element for technical abbreviations:

```html
<abbr title="Application Programming Interface">API</abbr>
```

## Use descriptive page titles

The page title (H1) should clearly describe the page content. This appears in browser tabs, search results, and screen reader announcements:

```yaml
---
title: "Configure authentication"
---
```

Avoid generic titles like "Configuration" or "Setup" without context.
