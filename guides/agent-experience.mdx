The agent experience encompasses how you interact with the Mintlify agent across different platforms and workflows. A well-designed agent experience helps your team maintain documentation efficiently while ensuring quality and consistency.

## Design your agent workflow

Consider how the agent fits into your team's documentation process.

### Choose your interaction method

The agent supports multiple interaction methods. Choose based on where your team already works:

- **Slack**: Best for collaborative documentation work and quick updates. Team members can discuss changes in threads and prompt the agent directly.
- **Linear**: Ideal for tracking documentation work alongside product development. Link documentation updates to feature releases.
- **Notion**: Useful for converting planning documents and specifications into documentation.
- **Dashboard**: Good for focused documentation work and reviewing agent capabilities.
- **API**: Best for automated workflows triggered by code changes or deployments.

### Establish clear responsibilities

Define who interacts with the agent and when:

- **Developers**: Prompt the agent to document new features or API changes from pull requests.
- **Technical writers**: Use the agent for content maintenance, restructuring, and quality improvements.
- **Product managers**: Convert product specs and requirements into user-facing documentation.
- **Support teams**: Update documentation based on common customer questions and issues.

## Set expectations with your team

Help your team understand what the agent can and cannot do.

### What the agent does well

The agent excels at:

- Updating existing documentation with new information from pull requests or specifications
- Maintaining consistency across multiple pages (renaming features, updating terminology)
- Converting technical information into user-friendly documentation
- Finding and fixing errors (typos, broken links, outdated examples)
- Generating first drafts from source material
- Coordinating changes across multiple files

### What requires human review

Always review agent output for:

- **Accuracy**: Verify technical details, especially for complex features or edge cases
- **Completeness**: Ensure all necessary information is included
- **User perspective**: Check that content addresses user needs and questions
- **Brand voice**: Confirm the tone matches your documentation standards
- **Context**: Validate that changes make sense within the broader documentation

## Optimize your prompts

The quality of agent output depends on the quality of your prompts.

### Provide context

Include relevant information in your prompts:

- Link to pull requests, Linear issues, or Slack threads with background information
- Attach files like diagrams, screenshots, or specifications
- Reference existing documentation pages that should be updated
- Mention your target audience and their technical level

### Be specific about outcomes

Focus on what users should be able to do after reading the documentation:

<CodeGroup>
```text Generic prompt
@mintlify Update the API docs
```

```text Specific prompt
@mintlify Update the API authentication docs to show developers how to use the new OAuth 2.0 flow. Include a complete example with token refresh.
```
</CodeGroup>

### Break down complex tasks

For large documentation projects, prompt the agent in stages:

1. **Structure**: `@mintlify Create an outline for a guide about setting up webhooks`
2. **Content**: `@mintlify Write the webhook setup guide based on this outline and the code in PR #123`
3. **Examples**: `@mintlify Add code examples for webhook signature verification in Python and Node.js`
4. **Polish**: `@mintlify Review the webhook guide and add troubleshooting steps for common issues`

## Integrate with your review process

The agent creates pull requests for all changes. Design a review process that maintains quality while moving quickly.

### Configure automerge for low-risk changes

Use [automerge](/guides/configure-automerge) for routine maintenance tasks:

- Fixing typos and formatting
- Updating version numbers
- Renaming features consistently across docs
- Fixing broken links

### Require review for content changes

Set up branch protection rules to require review for:

- New documentation pages
- Changes to getting started guides
- API reference updates
- Architecture or concept explanations

### Use draft PRs for work in progress

Set `asDraft: true` in API calls or mention "draft" in Slack prompts when you want to review changes before they're ready for merge.

## Customize agent behavior

Use an `AGENTS.md` file to ensure consistent output across all agent interactions.

### Document your standards

Include instructions for:

- **Code examples**: Programming languages, frameworks, and patterns to use
- **Terminology**: Product-specific terms and how to use them
- **Structure**: Required sections for different content types
- **Audience**: Technical level and background of your users

See [Customize agent behavior](/agent/customize) for detailed guidance and examples.

## Monitor and improve

Track how the agent performs and refine your approach.

### Review agent pull requests

Look for patterns in agent output:

- Are prompts consistently producing good results?
- Do certain types of tasks require more revision?
- Is the agent following your `AGENTS.md` instructions?

### Gather team feedback

Ask your team:

- Which agent interactions save the most time?
- Where does agent output need the most revision?
- What additional context would improve results?

### Iterate on your workflow

Adjust your agent experience based on what you learn:

- Update `AGENTS.md` with new standards and examples
- Refine prompt templates for common tasks
- Adjust automerge rules based on agent accuracy
- Add or remove interaction methods based on team usage

## Common patterns

### Feature release workflow

1. Developer opens PR with new feature code
2. Developer prompts agent in Slack: `@mintlify Document the new rate limiting feature based on PR #456`
3. Agent creates documentation PR
4. Technical writer reviews and approves
5. Documentation merges alongside feature release

### Support-driven updates

1. Support team identifies common customer question
2. Support team creates Linear issue: "Document how to handle webhook retries"
3. Agent creates documentation from Linear issue
4. Technical writer reviews and adds troubleshooting section
5. Support team links to new docs in responses

### Automated maintenance

1. Code changes push to main branch
2. GitHub Action triggers agent API call
3. Agent updates relevant documentation
4. Automerge merges low-risk changes
5. Team reviews content changes in weekly documentation review
