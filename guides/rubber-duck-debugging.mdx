Rubber duck debugging is a method of debugging code by articulating a problem in spoken or written natural language. The name is a reference to a story in *The Pragmatic Programmer* in which a developer would carry around a rubber duck and debug their code by forcing themselves to explain it, line by line, to the duck.

## Why it works

The act of teaching forces you to slow down and examine each assumption you've made. Most bugs hide in the gap between what you *think* the code does and what it *actually* does. Explaining things out loud surfaces that gap.

## How to do it

1. Find a rubber duck. Any inanimate object will do, but tradition favors the duck.
2. Place the duck somewhere it can "see" your screen.
3. Explain your goal in plain language: what are you trying to accomplish?
4. Walk through your code line by line, describing what each part does.
5. When you say something that doesn't match what the code actually does, you've found your bug.

## Variations

- **Rubber duck writing.** Explain your draft to the duck before publishing. Confusing sentences tend to reveal themselves.
- **Rubber duck planning.** Walk the duck through your architecture before you start building.
- **Rubber duck code review.** Read a teammate's PR aloud to the duck before approving it.

## A brief history

The earliest known reference is from Andy Hunt and Dave Thomas's *The Pragmatic Programmer* (1999). The technique predates the book in folklore — programmers have been talking to walls, plants, and cats for as long as there has been code to debug.
