Use the language switcher to create documentation in multiple languages. This guide explains how to structure files, configure navigation, and maintain translations effectively.

## File structure

Organize translated content in language-specific directories at the root of your documentation repository.

```
docs/
├── en/
│   ├── quickstart.mdx
│   ├── api/
│   │   └── introduction.mdx
│   └── guides/
│       └── getting-started.mdx
├── es/
│   ├── quickstart.mdx
│   ├── api/
│   │   └── introduction.mdx
│   └── guides/
│       └── getting-started.mdx
└── fr/
    ├── quickstart.mdx
    ├── api/
    │   └── introduction.mdx
    └── guides/
        └── getting-started.mdx
```

<Note>
  Keep the same file names and directory structure across all languages. This makes it easier to maintain translations and identify missing content.
</Note>

## Configure the language switcher

Add the `languages` array to your `navigation` object in `docs.json`. Each language requires a `language` field with an ISO 639-1 language code and can contain any other navigation fields.

```json docs.json
{
  "navigation": {
    "languages": [
      {
        "language": "en",
        "groups": [
          {
            "group": "Getting started",
            "pages": ["en/quickstart", "en/installation"]
          }
        ]
      },
      {
        "language": "es",
        "groups": [
          {
            "group": "Comenzar",
            "pages": ["es/quickstart", "es/installation"]
          }
        ]
      }
    ]
  }
}
```

## Language-specific navigation

Each language can have its own navigation structure, including translated group names, page titles, and descriptions.

```json docs.json
{
  "navigation": {
    "languages": [
      {
        "language": "en",
        "tabs": [
          {
            "tab": "Documentation",
            "groups": [
              {
                "group": "Getting started",
                "pages": ["en/index", "en/quickstart"]
              },
              {
                "group": "API reference",
                "pages": ["en/api/introduction"]
              }
            ]
          }
        ]
      },
      {
        "language": "fr",
        "tabs": [
          {
            "tab": "Documentation",
            "groups": [
              {
                "group": "Premiers pas",
                "pages": ["fr/index", "fr/quickstart"]
              },
              {
                "group": "Référence de l'API",
                "pages": ["fr/api/introduction"]
              }
            ]
          }
        ]
      }
    ]
  }
}
```

<Tip>
  Translate navigation labels (group names, tab names) to match the language of the content. This creates a fully localized experience for your users.
</Tip>

## Managing translations

### Translation workflow

1. **Create the source content:** Write and finalize content in your primary language.
2. **Duplicate the file structure:** Copy the directory structure to each language folder.
3. **Translate content:** Translate the content while preserving MDX components and code examples.
4. **Update navigation:** Add translated pages to the appropriate language section in `docs.json`.
5. **Review and test:** Verify translations and test the language switcher.

### Best practices

- **Maintain parity:** Keep the same pages across all languages. If a page doesn't have a translation yet, consider using the source language version temporarily with a note about the translation status.
- **Preserve code examples:** Keep code snippets, API endpoints, and technical terms consistent across languages. Only translate comments and explanatory text.
- **Use consistent terminology:** Create a glossary of technical terms and their translations to maintain consistency.
- **Version control:** Use Git branches to manage translation work separately from content updates.
- **Track translation status:** Maintain a spreadsheet or use Git issues to track which pages need translation or updates.

## Translated images and assets

### Image organization

Store language-specific images in corresponding directories:

```
public/
├── images/
│   ├── en/
│   │   └── dashboard-screenshot.png
│   ├── es/
│   │   └── dashboard-screenshot.png
│   └── fr/
│       └── dashboard-screenshot.png
```

Reference images using relative paths in your MDX files:

```mdx
<img src="/images/en/dashboard-screenshot.png" alt="Dashboard overview" />
```

### Shared assets

For images that don't contain text or language-specific content, use a shared directory:

```
public/
├── images/
│   ├── shared/
│   │   └── logo.png
│   ├── en/
│   │   └── ui-screenshot.png
│   └── es/
│       └── ui-screenshot.png
```

<Tip>
  Use shared assets whenever possible to reduce maintenance burden and storage requirements.
</Tip>

## SEO for multi-language sites

### Meta tags

Each translated page should have language-specific meta tags in the frontmatter:

```mdx
---
title: "Quickstart"
description: "Get started with our platform in minutes."
keywords: ["quickstart", "getting started", "tutorial"]
---
```

### Hreflang tags

Mintlify automatically generates hreflang tags for multi-language sites, helping search engines understand the relationship between translated pages.

### URL structure

Language-specific URLs follow the pattern:

```
https://docs.example.com/en/quickstart
https://docs.example.com/es/quickstart
https://docs.example.com/fr/quickstart
```

This structure helps search engines index content correctly and provides clear URLs for users.

### Sitemap

Mintlify automatically generates a sitemap that includes all language versions of your documentation. Search engines use this to discover and index your translated content.

## Testing your translations

Before deploying translated documentation:

1. **Verify the language switcher:** Ensure all languages appear in the dropdown and switch correctly.
2. **Check navigation:** Confirm that navigation labels are translated and pages load correctly.
3. **Test internal links:** Verify that links between pages work within each language.
4. **Review formatting:** Ensure MDX components render correctly in all languages.
5. **Validate images:** Check that language-specific images display properly.
6. **Test search:** Verify that search works within each language version.

## Common issues

### Missing translations

If a page exists in one language but not another, users will see a 404 error when switching languages. To prevent this:

- Maintain the same file structure across all languages
- Use placeholder pages with a translation notice if content isn't ready
- Track translation status to identify gaps

### Inconsistent navigation

Navigation structure should match across languages. If you add a new page or group to one language, add it to all languages to maintain consistency.

### Mixed language content

Avoid mixing languages within a single page. Keep all content, including navigation labels and component text, in the same language as the page content.
