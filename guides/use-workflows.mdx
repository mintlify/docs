Workflows run the agent automatically so your documentation stays accurate without manual effort. This guide covers how to choose the right workflow, configure it effectively, and write instructions that produce consistent results.

## When to use workflows

Workflows are best for tasks that are:

- **Repetitive** — The same task runs regularly on a predictable schedule or whenever a specific event occurs.
- **Time-consuming** — Tasks that take time to do manually but follow a consistent pattern.
- **Reactive** — Tasks that should happen automatically in response to code or content changes, like updating docs after a feature ships.

For one-off or exploratory tasks, use the agent directly in the [dashboard](https://app.mintlify.com) or from Slack.

## Choosing a trigger

Each workflow runs on one of three trigger types. Choose the one that matches when you need the work done.

| Trigger | Use when |
|---|---|
| Content update | The task should run whenever your documentation changes — translating new pages, fixing broken links, or auditing SEO metadata. |
| Code change | The task depends on what changed in source code — updating API references or identifying doc gaps after a PR merges. |
| Custom schedule | The task should run on a fixed cadence regardless of what changed — publishing weekly changelogs or running periodic grammar checks. |

## Predefined workflows

Mintlify includes predefined workflows for the most common documentation maintenance tasks. Toggle any of them on from the [Workflows](https://app.mintlify.com/products/workflows?tab=workflows) page.

### Codebase workflows

**Sync content with code** — Watches one or more source code repositories. When a pull request merges, the agent reviews the changes and updates any affected documentation. Best used with a review step so you can verify the agent's proposed changes before they go live.

**Changelog** — Drafts a changelog entry from recent product updates on a recurring schedule. Point it at your source code repositories and configure a weekly or monthly schedule.

**Improve content from assistant conversations** — Reviews recent trends in assistant questions to identify documentation gaps. Runs on a schedule and does not require source repositories.

### Maintenance workflows

**Translate content** — Keeps your translated languages in sync whenever content changes. Triggers on content updates and automatically creates translations for all target languages you configure.

**Fix broken links** — Finds and fixes broken internal and external links whenever your documentation changes.

**Audit SEO metadata** — Reviews titles, meta descriptions, headings, and canonical tags after content updates.

**Check grammar** — Finds and fixes typos, spelling mistakes, and grammar errors after content updates.

**Enforce writing style** — Applies your style guide's voice, tone, and rules to content changes.

## Custom workflows

Create a custom workflow for tasks that aren't covered by the predefined options. Custom workflows give you full control over the trigger, instructions, and context repositories.

See [Create a custom workflow](/workflows/create) for setup steps.

### Writing effective instructions

Custom workflow instructions are prompts the agent follows on every run. The quality of your instructions directly affects the consistency of results.

**Describe the outcome, not just the task.** Tell the agent what done looks like.

<CodeGroup>
```text Less effective
Review the API reference pages.
```

```text More effective
Review every page in the /api-reference directory. For each endpoint, confirm that the request parameters, response fields, and example payloads match the OpenAPI spec in the api/openapi.yaml file. Open a pull request with corrections for any discrepancies you find.
```
</CodeGroup>

**Write a procedure.** Agents follow step-by-step instructions more reliably than open-ended prompts.

```text Example procedure
1. Read the merged pull request diff from the trigger repository.
2. Identify any changed API endpoints, parameters, or response shapes.
3. Search the documentation for pages that reference those endpoints.
4. Update the affected pages to match the changes in the pull request.
5. Open a pull request with a summary of the pages you changed and why.
```

**Keep scope narrow.** A workflow that does one thing well is more reliable than one that tries to do many things. Split complex tasks across multiple workflows.

**Specify files and directories.** The more precise you are about where the agent should look, the less time it spends searching.

```text Example with specific paths
Check every MDX file in the /content directory for broken internal links (links beginning with /). 
For each broken link, either fix the path or remove the link if no replacement exists.
```

### Using context repositories

For custom workflows and some predefined workflows (Changelog, Sync content with code on a schedule), you can add context repositories the agent reads when the workflow runs.

Add context repositories when:
- Your documentation references code, APIs, or configuration that lives in a separate repository.
- You want the agent to compare your docs against source code when deciding what to update.
- The workflow prompt refers to files outside your documentation repository.

You can add up to 10 context repositories per workflow. Each GitHub repository must have the [Mintlify GitHub App](/deploy/github) installed.

### Update mode

Decide how much review you want before changes go live.

- **Automatic** — Changes merge directly into your documentation repository without review. Best for low-risk, high-frequency tasks like translation, grammar checks, and link fixes.
- **Require review** — The agent opens a pull request and a human reviews it before merging. Best for tasks that change content meaning, like syncing with code changes or updating API references.

## Reviewing and improving workflow results

Workflows run automatically, but they benefit from periodic review — especially when you first enable them.

1. Open the **Runs** tab on the [Workflows](https://app.mintlify.com/products/workflows) page to see recent run history and status.
2. Click a run to read the agent's summary and review any pull requests it opened.
3. If results are inconsistent, refine your instructions: add more specific procedures, narrow the file scope, or split the workflow into smaller tasks.

A few review cycles after enabling a new workflow are usually enough to dial in the instructions.
