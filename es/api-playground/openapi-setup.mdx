OpenAPI es una especificación para describir APIs. Mintlify soporta documentos OpenAPI 3.0+ para generar documentación interactiva de API y mantenerla actualizada.

## Agregar un archivo de especificación OpenAPI

Para documentar tus endpoints con OpenAPI, necesitas un documento OpenAPI válido en formato JSON o YAML que siga la [especificación OpenAPI 3.0+](https://swagger.io/specification/).

Puedes hacer referencia a múltiples especificaciones OpenAPI para crear páginas para tu API. Organiza secciones para cada especificación en tu navegación o haz referencia a endpoints específicos de diferentes especificaciones.

<Note>
  Mintlify soporta `$ref` para **referencias internas únicamente** dentro de un solo documento OpenAPI. Las referencias externas no son compatibles.
</Note>

### Describiendo tu API

Recomendamos los siguientes recursos para aprender y construir tus documentos OpenAPI.

* [Guía OpenAPI de Swagger](https://swagger.io/docs/specification/v3_0/basic-structure/) para aprender la sintaxis de OpenAPI.
* [Las fuentes Markdown de la especificación OpenAPI](https://github.com/OAI/OpenAPI-Specification/blob/main/versions/) para consultar detalles de la última especificación OpenAPI.
* [Editor Swagger](https://editor.swagger.io/) para editar, validar y depurar tu documento OpenAPI.
* [El CLI de Mint](https://www.npmjs.com/package/mint) para validar tu documento OpenAPI con el comando: `mint openapi-check <openapiFilenameOrUrl>`.

<Note>
  La Guía OpenAPI de Swagger es para OpenAPI v3.0, pero casi toda la información es aplicable a v3.1. Para más información sobre las diferencias entre v3.0 y v3.1, consulta [Migrando de OpenAPI 3.0 a 3.1.0](https://www.openapis.org/blog/2021/02/16/migrating-from-openapi-3-0-to-3-1-0)
  en el blog de OpenAPI.
</Note>

### Especificando la URL para tu API

Para habilitar las funciones de Mintlify como el playground de API, agrega un campo `servers` a tu documento OpenAPI con la URL base de tu API.

```json
{
  "servers": [
    {
      "url": "https://api.example.com/v1"
    }
  ]
}
```

En un documento OpenAPI, los diferentes endpoints de API se especifican por sus rutas, como `/users/{id}` o simplemente `/`. La URL base define dónde se deben añadir estas rutas. Para más información sobre cómo configurar el campo `servers`, consulta [API Server and Base Path](https://swagger.io/docs/specification/api-host-and-base-path/) en la documentación de OpenAPI.

El playground de API utiliza estas URLs de servidor para determinar dónde enviar las solicitudes. Si especificas múltiples servidores, un menú desplegable permitirá a los usuarios alternar entre servidores. Si no especificas un servidor, el playground de API utilizará el modo simple ya que no puede enviar solicitudes sin una URL base.

Si tu API tiene endpoints que existen en diferentes URLs, puedes [sobrescribir el campo server](https://swagger.io/docs/specification/v3_0/api-host-and-base-path/#overriding-servers) para una ruta u operación específica.

### Especificando la autenticación

Para habilitar la autenticación en tu documentación y playground de API, configura los campos `securitySchemes` y `security` en tu documento OpenAPI. Las descripciones de API y el Playground de API añadirán campos de autenticación basados en las configuraciones de seguridad en tu documento OpenAPI.

<Steps>
  <Step title="Define tu método de autenticación.">
    Añade un campo `securitySchemes` para definir cómo se autentican los usuarios.

    Este ejemplo muestra una configuración para autenticación bearer.

    ```json
    {
      "components": {
        "securitySchemes": {
          "bearerAuth": {
            "type": "http",
            "scheme": "bearer"
          }
        }
      }
    }
    ```
  </Step>

  <Step title="Aplica autenticación a tus endpoints.">
    Añade un campo `security` para requerir autenticación.

    ```json
    {
      "security": [
        {
          "bearerAuth": []
        }
      ]
    }
    ```
  </Step>
</Steps>

Los tipos de autenticación comunes incluyen:

* [API Keys](https://swagger.io/docs/specification/authentication/api-keys/): Para claves basadas en header, query o cookie.
* [Bearer](https://swagger.io/docs/specification/authentication/bearer-authentication/): Para tokens JWT u OAuth.
* [Basic](https://swagger.io/docs/specification/authentication/basic-authentication/): Para nombre de usuario y contraseña.

Si diferentes endpoints dentro de tu API requieren diferentes métodos de autenticación, puedes [sobrescribir el campo security](https://swagger.io/docs/specification/authentication/#:~:text=you%20can%20apply%20them%20to%20the%20whole%20API%20or%20individual%20operations%20by%20adding%20the%20security%20section%20on%20the%20root%20level%20or%20operation%20level%2C%20respectively.) para una operación específica.

Para más información sobre cómo definir y aplicar la autenticación, consulta [Authentication](https://swagger.io/docs/specification/authentication/) en la documentación de OpenAPI.

## `x-mint` extension

La extensión `x-mint` es una extensión personalizada de OpenAPI que proporciona control adicional sobre cómo se genera y muestra la documentación de tu API.

### Metadatos

Sobrescribe los metadatos predeterminados para las páginas de API generadas añadiendo `x-mint: metadata` a cualquier operación. Puedes usar cualquier campo de metadatos que sería válido en `MDX` frontmatter excepto por `openapi`:

```json {7-13}
{
  "paths": {
    "/users": {
      "get": {
        "summary": "Get users",
        "description": "Retrieve a list of users",
        "x-mint": {
          "metadata": {
            "title": "List all users",
            "description": "Fetch paginated user data with filtering options",
            "og:title": "Display a list of users"
          }
        },
        "parameters": [
          {
            // Parameter configuration
          }
        ]
      }
    }
  }
}
```

### Contenido

Añade contenido antes de la documentación de API autogenerada usando `x-mint: content`:

```json {6-8}
{
  "paths": {
    "/users": {
      "post": {
        "summary": "Create user",
        "x-mint": {
          "content": "## Prerequisites\n\nThis endpoint requires admin privileges and has rate limiting.\n\n<Note>User emails must be unique across the system.</Note>"
        },
        "parameters": [
          {
            // Parameter configuration
          }
        ]
      }
    }
  }
}
```

La extensión `content` soporta todos los componentes MDX y el formato de Mintlify.

### Href

Cambia la URL de la página del endpoint en tu documentación usando `x-mint: href`:

```json {6-8, 14-16}
{
  "paths": {
    "/legacy-endpoint": {
      "get": {
        "summary": "Legacy endpoint",
        "x-mint": {
          "href": "/deprecated-endpoints/legacy-endpoint"
        }
      }
    },
    "/documented-elsewhere": {
      "post": {
        "summary": "Special endpoint",
        "x-mint": {
          "href": "/guides/special-endpoint-guide"
        }
      }
    }
  }
}
```

Cuando `x-mint: href` está presente, la entrada de navegación enlazará directamente a la URL especificada en lugar de generar una página de API.

### MCP

Expone selectivamente endpoints como herramientas del Protocolo de Contexto de Modelo (MCP) usando `x-mint: mcp`. Solo habilita endpoints que sean seguros para el acceso público a través de herramientas de IA.

<ResponseField name="mcp" type="object">
  La configuración MCP para el endpoint.

  <Expandable title="MCP">
    <ResponseField name="enabled" type="boolean">
      Si se debe exponer el endpoint como una herramienta MCP. Tiene prioridad sobre la configuración a nivel de archivo.
    </ResponseField>

    <ResponseField name="name" type="string">
      El nombre de la herramienta MCP.
    </ResponseField>

    <ResponseField name="description" type="string">
      La descripción de la herramienta MCP.
    </ResponseField>
  </Expandable>
</ResponseField>

<CodeGroup>
  ```json Selective enablement {6-9} wrap
  {
    "paths": {
      "/users": {
        "post": {
          "summary": "Create user",
          "x-mint": {
            "mcp": {
              "enabled": true
            },
            // ...
          }
        }
      },
      "/users": {
        "delete": {
          "summary": "Delete user (admin only)",
          // No `x-mint: mcp` so this endpoint is not exposed as an MCP tool
          // ...
        }
      }
    }
  }
  ```

  ```json Global enablement {3-5, 9-13} wrap
  {
    "openapi": "3.1.0",
    "x-mcp": {
        "enabled": true // All endpoints are exposed as MCP tools by default
      },
    "paths": {
      "/api/admin/delete": {
        "delete": {
          "x-mint": {
            "mcp": {
              "enabled": false // Disable MCP for this endpoint
            }
          },
          "summary": "Delete resources"
        }
      }
    }
  }
  ```
</CodeGroup>

Para más información, consulta [Model Context Protocol](/es/ai/model-context-protocol).

## Auto-generar páginas de API

Añade un campo `openapi` a cualquier elemento de navegación en tu `docs.json` para generar automáticamente páginas para endpoints OpenAPI. Puedes controlar dónde aparecen estas páginas en tu estructura de navegación, como secciones de API dedicadas o con otras páginas.

El campo `openapi` acepta tanto una ruta de archivo en tu repositorio de documentación como una URL a un documento OpenAPI alojado.

Las páginas de endpoint generadas tienen estos valores de metadatos predeterminados:

* `title`: El campo `summary` de la operación, si está presente. Si no hay `summary`, el título se genera a partir del método HTTP y el endpoint.
* `description`: El campo `description` de la operación, si está presente.
* `version`: El valor `version` del ancla o pestaña padre, si está presente.
* `deprecated`: El campo `deprecated` de la operación. Si `true`, aparecerá una etiqueta de obsoleto junto al título del endpoint en la navegación lateral y en la página del endpoint.

<Tip>
  Para excluir endpoints específicos de tus páginas de API autogeneradas, añade la propiedad

  [x-hidden](/es/api-playground/managing-page-visibility#x-hidden)
  a la operación en tu especificación OpenAPI.
</Tip>

Hay dos enfoques para añadir páginas de endpoint a tu documentación:

1. **Secciones de API dedicadas**: Haz referencia a especificaciones OpenAPI en elementos de navegación para secciones de API dedicadas.
2. **Puntos finales selectivos**: Referencia a puntos finales específicos en tu navegación junto con otras páginas.

### Secciones de API dedicadas

Genera secciones de API dedicadas añadiendo un `openapi` campo a un elemento de navegación sin otras páginas. Se incluirán todos los puntos finales de la especificación:

```json {5}
"navigation": {
  "tabs": [
    {
        "tab": "API Reference",
        "openapi": "https://petstore3.swagger.io/api/v3/openapi.json"
    }
  ]
}
```

Para organizar múltiples especificaciones OpenAPI en secciones separadas de tu documentación, asigna cada especificación a un grupo diferente en tu jerarquía de navegación. Cada grupo puede hacer referencia a su propia especificación OpenAPI.

```json {8-11, 15-18}
"navigation": {
  "tabs": [
    {
      "tab": "API Reference",
      "groups": [
        {
          "group": "Users API",
          "openapi": {
            "source": "/path/to/users-openapi.json",
            "directory": "api-reference"
          }
        },
        {
          "group": "Admin API",
          "openapi": {
            "source": "/path/to/admin-openapi.json",
            "directory": "api-reference"
          }
        }
      ]
    }
  ]
}
```

<Note>
  El `directory` campo es opcional y especifica dónde se almacenan las páginas de API generadas en el repositorio de tu documentación. Si no se especifica, por defecto es el `api-reference`directorio de tu repositorio.
</Note>

### Puntos finales selectivos

Cuando quieres más control sobre dónde aparecen los puntos finales en tu documentación, puedes hacer referencia a puntos finales específicos en tu navegación. Este enfoque te permite generar páginas para puntos finales de API junto con otro contenido. También puedes usar este enfoque para mezclar puntos finales de diferentes especificaciones OpenAPI.

#### Establecer una especificación OpenAPI predeterminada

Configura una especificación OpenAPI predeterminada para un elemento de navegación. Luego haz referencia a puntos finales específicos en el `pages` field:

```json {12, 15-16}
"navigation": {
  "tabs": [
    {
      "tab": "Getting started",
      "pages": [
        "quickstart",
        "installation"
      ]
    },
    {
      "tab": "API reference",
      "openapi": "/path/to/openapi.json",
      "pages": [
        "api-overview",
        "GET /users",
        "POST /users",
        "guides/authentication"
      ]
    }
  ]
}
```

Cualquier entrada de página que coincida con el formato `METHOD /path` generará una página de API para ese punto final utilizando la especificación OpenAPI predeterminada.

#### Herencia de especificación OpenAPI

Las especificaciones OpenAPI se heredan en la jerarquía de navegación. Los elementos de navegación secundarios heredan la especificación OpenAPI de su padre a menos que definan la suya propia:

```json {3, 7-8, 11, 13-14}
{
  "group": "API reference",
  "openapi": "/path/to/openapi-v1.json",
  "pages": [
    "overview",
    "authentication",
    "GET /users",
    "POST /users",
    {
      "group": "Orders",
      "openapi": "/path/to/openapi-v2.json",
      "pages": [
        "GET /orders",
        "POST /orders"
      ]
    }
  ]
}
```

#### Puntos finales individuales

Haz referencia a puntos finales específicos sin establecer una especificación OpenAPI predeterminada incluyendo la ruta del archivo. Puedes hacer referencia a puntos finales de múltiples especificaciones OpenAPI en la misma sección de documentación.

```json {5-6}
"navigation": {
  "pages": [
    "introduction",
    "user-guides",
    "/path/to/users-openapi.json POST /users",
    "/path/to/orders-openapi.json GET /orders"
  ]
}
```

Este enfoque es útil cuando necesitas:

* Mezclar puntos finales de múltiples especificaciones OpenAPI dentro de la misma sección de navegación.
* Incluir solo puntos finales específicos de diferentes especificaciones en lugar de todos los puntos finales.
* Integrar puntos finales de API junto con otros tipos de páginas de documentación.

## Crear `MDX` archivos para páginas de API

Para tener control sobre páginas de puntos finales individuales, crea `MDX` páginas para cada operación. Esto te permite personalizar los metadatos de la página, añadir contenido, omitir ciertas operaciones o reordenar páginas en tu navegación a nivel de página.

Mira un [ejemplo de página MDX OpenAPI de MindsDB](https://github.com/mindsdb/mindsdb/blob/main/docs/rest/databases/create-databases.mdx?plain=1) y cómo aparece en su [documentación en vivo](https://docs.mindsdb.com/rest/databases/create-databases).

### Especificar archivos manualmente

Crea una `MDX` página para cada punto final y especifica qué operación OpenAPI mostrar usando el `openapi` campo en el frontmatter.

Cuando haces referencia a una operación OpenAPI de esta manera, el nombre, la descripción, los parámetros, las respuestas y el área de pruebas de API se generan automáticamente a partir de tu documento OpenAPI.

Si tienes múltiples archivos OpenAPI, incluye la ruta del archivo en tu referencia para asegurar que Mintlify encuentre el documento OpenAPI correcto. Si solo tienes un archivo OpenAPI, Mintlify lo detectará automáticamente.

<Note>
  Este enfoque funciona independientemente de si has establecido una especificación OpenAPI predeterminada en tu navegación. Puedes hacer referencia a cualquier punto final de cualquier especificación OpenAPI incluyendo la ruta del archivo en el frontmatter.
</Note>

Si quieres hacer referencia a un archivo OpenAPI externo, añade la URL del archivo a tu `docs.json`.

<CodeGroup>
  ```mdx Example
  ---
  title: "Get users"
  description: "Returns all plants from the system that the user has access to"
  openapi: "/path/to/openapi-1.json GET /users"
  deprecated: true
  version: "1.0"
  ---
  ```

  ```mdx Format
  ---
  title: "title of the page"
  description: "description of the page"
  openapi: openapi-file-path method path
  deprecated: boolean (not required)
  version: "version-string" (not required)
  ---
  ```
</CodeGroup>

<Note>
  El método y la ruta deben coincidir exactamente con la definición en tu especificación OpenAPI. Si el punto final no existe en el archivo OpenAPI, la página estará vacía.
</Note>

### Autogenerar `MDX` archivos

Usa nuestro [scraper](https://www.npmjs.com/package/@mintlify/scraping) de Mintlify para autogenerar `MDX` páginas para documentos OpenAPI grandes.

<Note>
  Tu documento OpenAPI debe ser válido o los archivos no se autogenerarán.
</Note>

El scraper genera:

* Una `MDX` página para cada operación en el `paths` campo de tu documento OpenAPI.
* Si tu documento OpenAPI es versión 3.1+, una `MDX` página para cada operación en el `webhooks` campo de tu documento OpenAPI.
* Un array de entradas de navegación que puedes añadir a tu `docs.json`.

<Steps>
  <Step title="Generar archivos `MDX`.">
    ```bash
    npx @mintlify/scraping@latest openapi-file <path-to-openapi-file>
    ```
  </Step>

  <Step title="Especificar una carpeta de salida.">
    ```bash
    npx @mintlify/scraping@latest openapi-file <path-to-openapi-file> -o api-reference
    ```

    Añade la `-o` bandera para especificar una carpeta donde colocar los archivos. Si no se especifica una carpeta, los archivos se colocarán en el directorio de trabajo.
  </Step>
</Steps>

### Crear `MDX` archivos para esquemas OpenAPI

Puedes crear páginas individuales para cualquier esquema OpenAPI definido en el `components.schemas` field:

<CodeGroup>
  ```mdx Example
  ---
  openapi-schema: OrderItem
  ---
  ```

  ```mdx Format
  ---
  openapi-schema: "schema-key"
  ---
  ```
</CodeGroup>

Si tienes esquemas con el mismo nombre en múltiples archivos, también puedes especificar el archivo OpenAPI:

<CodeGroup>
  ```mdx Example
  ---
  openapi-schema: en-schema.json OrderItem
  ---
  ```

  ```mdx Format
  ---
  openapi-schema: "path-to-schema-file schema-key"
  ---
  ```
</CodeGroup>

## Webhooks

Los webhooks son devoluciones de llamada HTTP que tu API envía para notificar a sistemas externos cuando ocurren eventos. Los webhooks son compatibles con documentos OpenAPI 3.1+.

### Definir webhooks en tu especificación OpenAPI

Añade un `webhooks` campo a tu documento OpenAPI junto con el `paths` campo.

Para más información sobre la definición de webhooks, consulta [Webhooks](https://spec.openapis.org/oas/v3.1.0#oasWebhooks) en la documentación de OpenAPI.

### Hacer referencia a webhooks en archivos MDX

Al crear páginas MDX para webhooks, usa `webhook` en lugar de métodos HTTP como `GET` o `POST`:

```mdx
---
title: "Example webhook"
description: "Triggered when an event occurs"
openapi: "path/to/openapi-file webhook example-webhook-name"
---
```

<Note>
  El nombre del webhook debe coincidir exactamente con la clave definida en el `webhooks` campo de tu especificación OpenAPI.
</Note>
