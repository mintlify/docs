OpenAPI es una especificación para describir APIs. Mintlify soporta documentos OpenAPI 3.0+ para generar documentación interactiva de API y mantenerla actualizada.

## Agregar un archivo de especificación OpenAPI

Para documentar tus endpoints con OpenAPI, necesitas una especificación OpenAPI válida en formato JSON o YAML que siga la [especificación OpenAPI 3.0+](https://swagger.io/specification/).

Puedes hacer referencia a múltiples especificaciones OpenAPI para crear páginas para tu API. Organiza secciones para cada especificación en tu navegación o haz referencia a endpoints específicos de diferentes especificaciones.

<Note>
  Mintlify soporta `$ref` para **referencias internas únicamente** dentro de un solo documento OpenAPI. Las referencias externas no son compatibles.
</Note>

### Describe tu API

Recomendamos los siguientes recursos para aprender y construir tu especificación OpenAPI.

* [Guía OpenAPI de Swagger](https://swagger.io/docs/specification/v3_0/basic-structure/) para aprender la sintaxis de OpenAPI.
* [Las fuentes Markdown de la especificación OpenAPI](https://github.com/OAI/OpenAPI-Specification/blob/main/versions/) para consultar detalles de la última especificación OpenAPI.
* [Editor Swagger](https://editor.swagger.io/) para editar, validar y depurar tu documento OpenAPI.
* [El CLI de Mint](https://www.npmjs.com/package/mint) para validar tu documento OpenAPI con el comando: `mint openapi-check <openapiFilenameOrUrl>`.

<Note>
  La Guía OpenAPI de Swagger es para OpenAPI v3.0, pero casi toda la información es aplicable a v3.1. Para más información sobre las diferencias entre v3.0 y v3.1, consulta [Migrando de OpenAPI 3.0 a 3.1.0](https://www.openapis.org/blog/2021/02/16/migrating-from-openapi-3-0-to-3-1-0) en el blog de OpenAPI.
</Note>

### Especificar la URL base para tu API

Para habilitar el playground de API, agrega un campo `servers` a tu especificación OpenAPI con la URL base de tu API.

```json
{
  "servers": [
    {
      "url": "https://api.example.com/v1"
    }
  ]
}
```

En una especificación OpenAPI, los diferentes endpoints de API se especifican por sus rutas, como `/users/{id}` o simplemente `/`. La URL base define dónde se deben añadir estas rutas. Para más información sobre cómo configurar el campo `servers`, consulta [Servidor API y Ruta Base](https://swagger.io/docs/specification/api-host-and-base-path/) en la documentación de OpenAPI.

El playground de API utiliza estas URLs de servidor para determinar dónde enviar las solicitudes. Si especificas múltiples servidores, un menú desplegable permite a los usuarios alternar entre servidores. Si no especificas un servidor, el playground de API utiliza el modo simple ya que no puede enviar solicitudes sin una URL base.

Si tu API tiene endpoints que existen en diferentes URLs, puedes [sobrescribir el campo `servers`servers](https://swagger.io/docs/specification/v3_0/api-host-and-base-path/#overriding-servers) para una ruta u operación específica.

### Especificar autenticación

Para habilitar la autenticación en tu documentación y playground de API, configura los campos `securitySchemes`securitySchemes`security` y&#x20;

<Steps>
  <Step title="Define tu método de autenticación.">
    Añade un campo `securitySchemes`securitySchemes

    Este ejemplo muestra una configuración para autenticación bearer.

    ```json
    {
      "components": {
        "securitySchemes": {
          "bearerAuth": {
            "type": "http",
            "scheme": "bearer"
          }
        }
      }
    }
    ```
  </Step>

  <Step title="Aplica autenticación a tus endpoints.">
    Añade un campo `security`security

    ```json
    {
      "security": [
        {
          "bearerAuth": []
        }
      ]
    }
    ```
  </Step>
</Steps>

Los tipos de autenticación comunes incluyen:

* [API Keys](https://swagger.io/docs/specification/authentication/api-keys/): Para claves basadas en encabezados, consultas o cookies.
* [Bearer](https://swagger.io/docs/specification/authentication/bearer-authentication/): Para tokens JWT u OAuth.
* [Basic](https://swagger.io/docs/specification/authentication/basic-authentication/): Para nombre de usuario y contraseña.

Si diferentes endpoints dentro de tu API requieren diferentes métodos de autenticación, puedes [sobrescribir el campo `security`security](https://swagger.io/docs/specification/authentication/#:~:text=you%20can%20apply%20them%20to%20the%20whole%20API%20or%20individual%20operations%20by%20adding%20the%20security%20section%20on%20the%20root%20level%20or%20operation%20level%2C%20respectively.) para una operación específica.

Para más información sobre cómo definir y aplicar la autenticación, consulta [Authentication](https://swagger.io/docs/specification/authentication/) en la documentación de OpenAPI.

## Personalizar tus páginas de endpoints

Personaliza tus páginas de endpoints añadiendo la extensión `x-mint`x-mintlify`x-mint` a tu especificación OpenAPI. La extensión&#x20;

### Metadatos

Sobrescribe los metadatos predeterminados para las páginas de API generadas añadiendo `x-mint: metadata`x-mintlify.metadata`openapi`.

```json {7-13}
{
  "paths": {
    "/users": {
      "get": {
        "summary": "Get users",
        "description": "Retrieve a list of users",
        "x-mint": {
          "metadata": {
            "title": "List all users",
            "description": "Fetch paginated user data with filtering options",
            "og:title": "Display a list of users"
          }
        },
        "parameters": [
          {
            // Parameter configuration
          }
        ]
      }
    }
  }
}
```

### Contenido

Añade contenido antes de la documentación de API autogenerada usando `x-mint: content`x-mintlify.content`x-mint: content`. La extensión&#x20;

```json {6-8}
{
  "paths": {
    "/users": {
      "post": {
        "summary": "Create user",
        "x-mint": {
          "content": "## Prerequisites\n\nThis endpoint requires admin privileges and has rate limiting.\n\n<Note>User emails must be unique across the system.</Note>"
        },
        "parameters": [
          {
            // Parameter configuration
          }
        ]
      }
    }
  }
}
```

### Href

Cambia la URL de la página del endpoint en tu documentación usando `x-mint: href`x-mintlify.href`x-mint: href`. Cuando&#x20;

```json {6-8, 14-16}
{
  "paths": {
    "/legacy-endpoint": {
      "get": {
        "summary": "Legacy endpoint",
        "x-mint": {
          "href": "/deprecated-endpoints/legacy-endpoint"
        }
      }
    },
    "/documented-elsewhere": {
      "post": {
        "summary": "Special endpoint",
        "x-mint": {
          "href": "/guides/special-endpoint-guide"
        }
      }
    }
  }
}
```

## Auto-generar páginas de API

Añade un campo `openapi`openapi`docs.json` a cualquier elemento de navegación en tu&#x20;

El campo `openapi`openapi

Las páginas de endpoints generadas tienen estos valores de metadatos predeterminados:

* `title`title`summary`summary`summary`summary
* `description`description`description`description
* `version`version`version`version
* `deprecated`deprecated`deprecated`deprecated: true`true`deprecated: true

<Tip>
  Para excluir endpoints específicos de tus páginas de API autogeneradas, añade la propiedad [x-hidden](/es/api-playground/managing-page-visibility#x-hidden) a la operación en tu especificación OpenAPI.
</Tip>

Hay dos enfoques para añadir páginas de endpoints a tu documentación:

1. **Secciones de API dedicadas**: Referencia especificaciones OpenAPI en elementos de navegación para secciones de API dedicadas.
2. **Endpoints selectivos**: Referencia endpoints específicos en tu navegación junto con otras páginas.

### Secciones de API dedicadas

Genera secciones de API dedicadas añadiendo un campo `openapi`openapi

```json {5}
"navigation": {
  "tabs": [
    {
        "tab": "API Reference",
        "openapi": "https://petstore3.swagger.io/api/v3/openapi.json"
    }
  ]
}
```

Para organizar múltiples especificaciones OpenAPI en secciones separadas de tu documentación, asigna cada especificación a un grupo diferente en tu jerarquía de navegación. Cada grupo puede hacer referencia a su propia especificación OpenAPI.

```json {8-11, 15-18}
"navigation": {
  "tabs": [
    {
      "tab": "API Reference",
      "groups": [
        {
          "group": "Users API",
          "openapi": {
            "source": "/path/to/users-openapi.json",
            "directory": "users-api-reference"
          }
        },
        {
          "group": "Admin API",
          "openapi": {
            "source": "/path/to/admin-openapi.json",
            "directory": "admin-api-reference"
          }
        }
      ]
    }
  ]
}
```

<Note>
  El `directory`field is optional and specifies where generated API pages are stored in your docs repo. If not specified, defaults to the `api-reference`directory of your repo.
</Note>

### Endpoints selectivos

Cuando deseas tener más control sobre dónde aparecen los endpoints en tu documentación, puedes hacer referencia a endpoints específicos en tu navegación. Este enfoque te permite generar páginas para endpoints de API junto con otro contenido. También puedes usar este enfoque para mezclar endpoints de diferentes especificaciones OpenAPI.

#### Establecer una especificación OpenAPI predeterminada

Configura una especificación OpenAPI predeterminada para un elemento de navegación. Luego haz referencia a endpoints específicos en el campo `pages`.

```json {12, 15-16}
"navigation": {
  "tabs": [
    {
      "tab": "Getting started",
      "pages": [
        "quickstart",
        "installation"
      ]
    },
    {
      "tab": "API reference",
      "openapi": "/path/to/openapi.json",
      "pages": [
        "api-overview",
        "GET /users",
        "POST /users",
        "guides/authentication"
      ]
    }
  ]
}
```

Cualquier entrada de página que coincida con el formato `METHOD /path` genera una página de API para ese endpoint utilizando la especificación OpenAPI predeterminada.

#### Herencia de especificación OpenAPI

Las especificaciones OpenAPI se heredan en la jerarquía de navegación. Los elementos de navegación secundarios heredan la especificación OpenAPI de su padre a menos que definan la suya propia.

```json {3, 7-8, 11, 13-14}
{
  "group": "API reference",
  "openapi": "/path/to/openapi-v1.json",
  "pages": [
    "overview",
    "authentication",
    "GET /users",
    "POST /users",
    {
      "group": "Orders",
      "openapi": "/path/to/openapi-v2.json",
      "pages": [
        "GET /orders",
        "POST /orders"
      ]
    }
  ]
}
```

#### Endpoints individuales

Haz referencia a endpoints específicos sin establecer una especificación OpenAPI predeterminada incluyendo la ruta del archivo. Puedes hacer referencia a endpoints de múltiples especificaciones OpenAPI en la misma sección de documentación.

```json {5-6}
"navigation": {
  "pages": [
    "introduction",
    "user-guides",
    "/path/to/users-openapi.json POST /users",
    "/path/to/orders-openapi.json GET /orders"
  ]
}
```

Este enfoque es útil cuando necesitas endpoints individuales de diferentes especificaciones, solo quieres incluir endpoints seleccionados, o quieres incluir endpoints junto con otros tipos de documentación.

## Crear páginas MDX a partir de tu especificación OpenAPI

Para un control más granular sobre páginas de endpoints individuales, crea páginas MDX a partir de tu especificación OpenAPI. Esto te permite personalizar los metadatos de la página, el contenido, y reordenar o excluir páginas en tu navegación mientras sigues utilizando los parámetros y respuestas generados automáticamente.

Hay dos formas de documentar tu especificación OpenAPI con páginas MDX individuales:

* Documentar endpoints con el campo `openapi` en el frontmatter.
* Documentar modelos de datos con el campo `openapi-schema` en el frontmatter.

### Documentar endpoints

Crea una página para cada endpoint y especifica qué operación OpenAPI mostrar usando el campo `openapi` en el frontmatter.

<CodeGroup>
  ```mdx Example
  ---
  title: "Get users"
  description: "Returns all plants from the system that the user has access to"
  openapi: "/path/to/openapi-1.json GET /users"
  deprecated: true
  version: "1.0"
  ---
  ```

  ```mdx Format
  ---
  title: "title of the page"
  description: "description of the page"
  openapi: openapi-file-path method path
  deprecated: boolean (not required)
  version: "version-string" (not required)
  ---
  ```
</CodeGroup>

El método y la ruta deben coincidir exactamente con tu especificación OpenAPI. Si tienes múltiples especificaciones OpenAPI, incluye la ruta del archivo en tu referencia. Las URLs OpenAPI externas pueden ser referenciadas en `docs.json`.

#### Autogenerar páginas de endpoints

Para autogenerar archivos MDX a partir de tu especificación OpenAPI, usa el [scraper](https://www.npmjs.com/package/@mintlify/scraping) de Mintlify.

```bash
npx @mintlify/scraping@latest openapi-file <path-to-openapi-file> -o <folder-name>
```

<Tip>
  Añade la bandera `-o` para especificar una carpeta donde colocar los archivos. Si no se especifica una carpeta, los archivos se colocarán en el directorio de trabajo.
</Tip>

### Documentar modelos de datos

Crea una página para cada estructura de datos definida en `components.schemas` de tu especificación OpenAPI usando el campo `openapi-schema` en el frontmatter.

<CodeGroup>
  ```mdx Example
  ---
  openapi-schema: OrderItem
  ---
  ```

  ```mdx Format
  ---
  openapi-schema: "openapi-file-path schema-key"
  ---
  ```
</CodeGroup>

Si tienes esquemas con el mismo nombre en múltiples archivos, especifica el archivo OpenAPI:

<CodeGroup>
  ```mdx Example
  ---
  openapi-schema: en-schema.json OrderItem
  ---
  ```

  ```mdx Format
  ---
  openapi-schema: "path-to-schema-file schema-key"
  ---
  ```
</CodeGroup>

## Webhooks

Los webhooks son callbacks HTTP que tu API envía para notificar a sistemas externos cuando ocurren eventos. Los webhooks son compatibles con documentos OpenAPI 3.1+.

Añade un campo `webhooks` a tu documento OpenAPI junto con el campo `paths`.

Para más información sobre la definición de webhooks, consulta [Webhooks](https://spec.openapis.org/oas/v3.1.0#oasWebhooks) en la documentación de OpenAPI.

Para crear una página MDX para un webhook (OpenAPI 3.1+), usa `webhook` en lugar de un método HTTP:

```mdx
---
title: "Order updated webhook"
description: "Triggered when an order is updated"
openapi: "openapi.json webhook orderUpdated"
---
```

El nombre del webhook debe coincidir exactamente con la clave en el campo `webhooks` de tu especificación OpenAPI.
