El editor web tiene dos capas de configuración:

- **Su configuración** se aplica solo a usted y controla cómo la IA del editor le ayuda con sus ediciones.
- **La configuración de publicación** se aplica a todos en un deployment y determina qué sucede cuando confirma cambios y los convierte en pull requests.

Puede configurar ambas desde el panel de configuración. Para abrirlo, haga clic en el icono <Icon icon="settings" /> de configuración en la barra de herramientas del editor.

<div id="ai-instructions">

## Instrucciones de IA

</div>

Las instrucciones de IA son pautas persistentes que el editor envía a la IA junto con su solicitud. Úselas para capturar reglas de estilo y tono que no quiere repetir cada vez, como voz, terminología o convenciones de formato.

Sus instrucciones se aplican a:

- Acciones **Edit with AI** sobre una selección, como reescribir, ampliar o corregir.
- **Sesiones del agente** iniciadas desde el editor.

Cada compañero de equipo mantiene sus propias instrucciones, asociadas a su cuenta de usuario.

<div id="when-to-use-ai-instructions">

### Cuándo usar las instrucciones de IA

</div>

Añada instrucciones de IA cuando se descubra repitiendo las mismas pautas en los prompts, por ejemplo:

- Imponer el uso de la segunda persona o encabezados en mayúscula inicial.
- Preferir nombres de producto o terminología específicos.
- Prohibir lenguaje de marketing o frases de relleno.
- Exigir ciertos componentes, como usar siempre `<Note>` para los avisos.

Mantenga las instrucciones cortas y específicas. La IA las sigue en cada solicitud, así que las reglas vagas o contradictorias degradan los resultados.

<div id="configure-ai-instructions">

### Configurar las instrucciones de IA

</div>

1. Haga clic en el icono <Icon icon="settings" /> de configuración en la barra de herramientas del editor.
2. En el campo **AI instructions**, introduzca las pautas que quiere que siga la IA.
3. Guarde sus cambios.

Ejemplo:

```text
- Use second person ("you") and active voice.
- Use sentence case for all headings.
- Refer to the product as "Acme" — never "Acme Inc." or "the platform".
- Wrap notes and warnings in <Note> or <Warning> components.
- Do not add introductory filler like "In this guide" or "Let's explore".
```

Deje el campo vacío para eliminar sus instrucciones.

<div id="publishing-settings">

## Configuración de publicación

</div>

La configuración de publicación se establece por deployment y se aplica a todas las personas que publican desde el editor. Controla cómo el editor genera, abre y fusiona pull requests y commits.

Necesita acceso de administrador a su deployment de Mintlify para cambiar la configuración de publicación.

<div id="pull-request-instructions">

### Instrucciones para el pull request

</div>

Las instrucciones para el pull request guían a la IA cuando genera el título y la descripción de un pull request. Se aplican siempre que el editor abre un pull request en su nombre, incluyendo los flujos **Create pull request** y **Merge and publish**.

Use las instrucciones para el pull request para estandarizar lo que ven los revisores, por ejemplo:

- Secciones obligatorias como **Summary** y **Changes**.
- Una plantilla de descripción que enlace con un sistema de seguimiento.
- Requisitos de tono o de longitud para los títulos.

Ejemplo:

```text
Title: imperative mood, under 70 characters, no trailing period.
Description: include a "## Summary" section (one sentence) and a
"## Changes" section as a bulleted list. Link any referenced page
using its relative path.
```

<div id="create-pull-requests-as-drafts-by-default">

### Crear pull requests como borradores por defecto

</div>

Active esta opción para que el editor abra todos los pull requests nuevos en estado de borrador. No puede fusionar un pull request en borrador hasta que lo marque como listo para revisión. Esto es útil cuando:

- Su equipo requiere una revisión manual antes de abrir el pull request para aprobación.
- Quiere compartir URLs de vista previa sin indicar que el cambio está listo para fusionarse.

Todavía puede marcar un pull request como listo para revisión desde su proveedor de Git.

<div id="default-merge-method">

### Método de fusión predeterminado

</div>

Elija cómo fusiona el editor los pull requests cuando hace clic en **Merge and publish**:

- **Merge**: Crea un commit de fusión que preserva el historial completo de la branch.
- **Squash**: Combina todos los commits de la branch en un único commit en su deployment branch.
- **Rebase**: Reproduce cada commit de la branch sobre su deployment branch sin crear un commit de fusión.

El editor usa el método seleccionado por defecto. Si pasa un método de fusión explícito a través de la API o de la interfaz de su proveedor de Git, esa elección tiene prioridad.

<Tip>
  Ajuste su método de fusión predeterminado a las reglas de protección de branches de su proveedor de Git. Si su deployment branch solo permite fusiones tipo squash, configure el valor predeterminado en **Squash** para evitar fusiones fallidas desde el editor.
</Tip>
