Docs-as-code is a way of managing documentation using the same tools and workflows developers use for software: version control, branches, pull requests, and code review.

Mintlify stores your documentation in a Git repository. The web editor gives writers a browser-based way to work with that repository without needing to know Git commands. This means writers and developers use the same underlying system, which makes it straightforward to collaborate, review changes, and keep documentation in sync with your product.

## What Git does for your docs

Git tracks every change made to your documentation: what changed, who changed it, and when. This means you can:

- See the full history of any page.
- Revert to a previous version.
- Work on updates in isolation without affecting your live site.
- Review changes before they go live.

Your documentation repository is the collection of files—and their history—that makes up your documentation site.

## How saving and publishing work

In the web editor, saving and publishing are separate steps.

**Saving** happens automatically as you type. Your edits are stored on Mintlify's servers and persist across browser tabs, devices, and network interruptions. No action required.

**Publishing** commits your changes to your Git repository. When you click **Publish** or **Save in branch**, the editor writes your content as MDX files and creates a Git commit. Your live site updates after the build completes.

This separation keeps your Git history clean. You can edit freely, and only deliberate publish actions create commits.

## Automatic sync with Git

When someone pushes changes to your repository from outside the editor—from the CLI, a merged pull request, or any other Git client—the editor incorporates those changes automatically using a three-way merge.

The merge compares three versions:

- **Base**: The last version both the editor and Git agreed on.
- **Local**: Your current editor content, including unsaved edits.
- **Remote**: The newly pushed version from Git.

Non-overlapping changes apply automatically. If a remote change and your local edit affect the same part of a page, the editor highlights the conflict so you can resolve it.

<Note>
  Commits made from the web editor are recognized automatically and don't trigger a sync back into the editor.
</Note>
