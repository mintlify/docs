## What it is

Workflows let customers define automated agent tasks for their docs. Instead of manually updating docs when code ships, they define a trigger (PR merge, cron schedule), write instructions in plain English, and the Mintlify agent handles the rest: reading their codebase, writing or updating content, validating the build, and opening a PR.

Everything is configured in a markdown file inside their docs repo (`.mintlify/workflows/` directory) or from the dashboard. Version-controlled, reviewable, fits into the git workflow they already use.

## What problem it solves

Docs drift out of date as teams ship faster. Changelogs get skipped, broken links pile up, translations fall behind, and new features go undocumented. Nobody owns recurring maintenance, so it slips through the cracks.

Workflows turn these recurring tasks into reliable automated systems.

## Typical use cases

| Use case | Trigger | What the agent does |
|----------|---------|-------------------|
| **Changelog generation** | PR merge or weekly cron | Reviews merged PRs, writes changelog entries |
| **Draft docs for new features** | PR merge to product repo | Identifies new features, drafts documentation |
| **Style and quality audit** | Daily cron | Checks for broken links, grammar, style violations |
| **Translation sync** | Weekly cron | Compares English source against translations, flags or updates stale pages |
| **API reference updates** | PR merge to product repo | Detects endpoint/parameter changes, updates API docs |

## How it works (for demos)

1. **Create a workflow** - Either from the dashboard (click New Workflow) or by adding a `.md` file to `.mintlify/workflows/` in the docs repo
2. **Define the trigger** - Cron schedule (`"0 9 * * 1"` = every Monday 9am UTC) or push events on specific repos/branches
3. **Write the prompt** - Plain English instructions for what the agent should do
4. **Add context repos** - Give the agent read access to product code repos so it has full context
5. **Set automerge** - `false` opens a PR for review (default), `true` commits directly

When triggered, the agent spins up an isolated sandbox, clones the repos, follows the prompt, validates the build, and takes action.

## What's included

- Up to 10 workflow definitions (Enterprise)
- Up to 20 runs per day per workflow
- Push triggers (PR merge, direct push) and cron triggers
- Context repos for cross-repo awareness
- PR or direct commit output
- Dashboard for creating and monitoring workflows
- Free during beta for Enterprise customers

## What's NOT included

- **No Slack or API triggers yet** - Only cron and push events right now
- **No chaining** - Workflows are independent, you can't trigger one workflow from another
- **No external network access** - The agent runs in a sandbox and can't hit external APIs or install packages
- **No interactive approval steps** - It's fully automated once triggered, no human-in-the-loop during execution
- **No dashboard editing of prompts yet** - Creation works from dashboard, but editing still requires updating the file in the repo
- **Pro plan availability is limited** - Full self-serve coming soon

## Positioning and talk track

### The pitch (30 seconds)

"Your docs are a living product. Every time your team ships, your docs should keep up. Workflows let you define exactly when our AI agent takes action: when a PR merges, on a schedule, whatever you need. The agent reads your codebase, updates your docs, and opens a PR for review. It's automation defined in your repo, version-controlled, on your terms."

### For engineering leaders

Lead with developer velocity. "Your engineers shouldn't be spending time updating docs after every PR. Workflows handle changelogs, API reference updates, and feature docs automatically."

### For technical writers and DevRel

Do NOT position as replacing them. Lead with augmentation and observability. "Workflows handle the grunt work: link checks, translation syncs, changelog generation. You focus on content strategy and quality."

### For founders (Series A-C)

Lead with speed. "You're shipping fast, your docs should keep up without adding headcount."

## Objection handling

| Objection | Response |
|-----------|----------|
| "We don't trust AI to write our docs" | "That's why automerge is off by default. Every change opens a PR for your team to review. Think of it as a first draft from someone who read the entire codebase." |
| "Our docs setup is too complex" | "Workflows work with whatever you have. The agent gets full context on your docs and code repos. It follows the same conventions your team uses." |
| "We already have CI/CD for this" | "CI can catch broken builds, but it can't write a changelog entry or draft docs for a new feature. Workflows handle the content work, not the build process." |
| "What if the agent makes bad changes?" | "It opens a PR. Your team reviews it like any other PR. Bad changes get caught the same way they always do." |
| "Suggestions worked fine for us" | "Workflows give you the same agent intelligence with full control over timing and output. You pick the trigger, the instructions, and whether it opens a PR or commits directly." |
