You have multiple ways to start a conversation with the assistant. Each method opens a chat panel on the right side of your docs. You can ask any question and the assistant searches your documentation for an answer. If the assistant cannot retrieve relevant information, it responds that it cannot answer the question.

## Use the assistant in local preview

The assistant is available in your local preview when you log in with the CLI. This lets you test assistant behavior, custom instructions in `Assistant.md`, and responses while developing documentation locally.

1. If you haven't already, install the CLI with `npm i -g mint`.
2. Run `mint login` to authenticate.
3. Run `mint dev` to start the local preview.

The assistant in local preview uses the same indexed content as your deployed documentation site.

## Keyboard shortcut

Open the assistant chat panel with the keyboard shortcut <kbd>Command</kbd> + <kbd>I</kbd> on macOS and <kbd>Ctrl</kbd> + <kbd>I</kbd> on Windows.

## URL parameters

Open the assistant from any page on your documentation site by adding the `?assistant` query parameter to the URL. Use this to deep-link to the assistant from external content like emails, support replies, in-app banners, or onboarding flows.

- `?assistant`: Open the assistant chat panel without sending a message.
- `?assistant=<message>`: Open the assistant chat panel and submit `<message>` as the user's first message. URL-encode the value so spaces and special characters survive the request.

```text Examples
https://yourdocs.mintlify.site/quickstart?assistant
https://yourdocs.mintlify.site/quickstart?assistant=How%20do%20I%20get%20started%3F
```

## File attachments

Attach files to assistant messages to provide additional context.

Supported file types:
- **Images**: JPEG, PNG, GIF, WebP, SVG
- **Documents**: PDF
- **Code and text files**: JavaScript (`.js`, `.jsx`, `.mjs`, `.cjs`), TypeScript (`.ts`, `.tsx`), Python, HTML, CSS, Markdown, MDX, JSON, YAML, XML, SQL, CSV, plain text, shell scripts (`.sh`, `.bash`, `.env`), GraphQL, TOML, Go, Rust, Ruby, Java, Kotlin, Swift, C (`.c`, `.h`), C++ (`.cpp`, `.hpp`), C#, PHP, Lua, R, Scala

Limits:
- Maximum file size: 5 MB per attachment
- Maximum attachments per message: 10

## Troubleshooting

<Accordion title="Assistant chat bar not visible">
  If the assistant UI is not visible in specific browsers, you may need to submit a false positive report to [EasyList](https://easylist.to). Browsers that use the EasyList Cookies List like Brave and Comet sometimes block the assistant or other UI elements. The EasyList Cookies List includes a domain-specific rule that hides fixed elements on certain domains to block cookie banners. This rule inadvertently affects legitimate UI components.

  Submit a false positive report to [EasyList](https://github.com/easylist/easylist) to request removal of the rule. This resolves the issue for all users once the filter list updates.

</Accordion>

<Accordion title="Assistant fails to respond and shows a Request ID">
  If the assistant returns an error like `Something went wrong. Request ID: ...`, the request reached Mintlify but failed on its way back to the browser, or a network layer between the browser and Mintlify blocked it. The `Request ID` is an opaque identifier the Mintlify support team can use to look up server-side logs for that specific request. Copy it into your support message so the team can trace the failure.

  Before contacting support, try each of the following:

  - Reload the page and retry the request from a fresh session.
  - Retry from another network (for example, a personal hotspot) to isolate whether a corporate proxy, VPN, or firewall is interfering.
  - Retry in a different browser or a private window with extensions disabled.
  - If you use Brave, lower **Shields** for the docs site. Aggressive **Trackers & ads blocking** and **Block scripts** settings can cancel the assistant's streaming request.
  - Disable ad-blockers and privacy extensions (for example, uBlock Origin, AdGuard, Ghostery) for the docs site. Filter lists occasionally flag legitimate Mintlify or analytics hostnames.

  If the failure is reproducible on multiple networks and in a clean browser, share the `Request ID`, the docs URL, and the approximate time of the failure with support so they can locate the corresponding server-side log.
</Accordion>

<Accordion title="Assistant, search, or MCP blocked by a corporate network">
  The assistant, search, and the docs [MCP server](/ai/model-context-protocol) all rely on outbound requests from the browser to Mintlify. Corporate proxies, VPNs, Zscaler-style secure web gateways, and SSE-inspecting firewalls sometimes break these requests, most often by:

  - Terminating long-lived streaming responses (the assistant uses server-sent events for token-by-token replies).
  - Stripping or rewriting response headers required for streaming.
  - Blocking requests to Mintlify or CDN hostnames outright.
  - Injecting a Content Security Policy that removes required domains. See [CSP configuration](/deploy/csp-configuration) for the full list.

  Ask your network administrator to allow the following hostnames from end-user browsers on the docs site:

  | Hostname | Purpose |
  |:---------|:--------|
  | Your docs domain (for example, `docs.example.com` and `*.mintlify.app`) | Serves the docs site and proxies assistant, search, and MCP requests |
  | `leaves.mintlify.com` | Mintlify Assistant API |
  | `*.mintlify.com` | Dashboard, API, and analytics proxy |
  | `*.mintlify.site` | Preview and published docs content |
  | `mintcdn.com`, `*.mintcdn.com` | Images, favicons, and other static assets |
  | `d1ctpt7j8wusba.cloudfront.net` | Mint version and release metadata |
  | `d3gk2c5xim1je2.cloudfront.net` | Icons, images, and logos |
  | `d4tuoctqmanu0.cloudfront.net` | KaTeX CSS and fonts |
  | `hcaptcha.com`, `*.hcaptcha.com` | hCaptcha verification for assistant requests |

  In addition, the connection must allow streaming responses (`text/event-stream`) to pass through without buffering. Some inspection proxies buffer streaming responses until the connection closes, which makes the assistant appear to hang and eventually fail.
</Accordion>

<Accordion title="Self-check for blocked assistant requests">
  If the assistant, search, or MCP is failing and you suspect the visitor's network is at fault, run this quick check from the same browser and network where the failure occurred:

  1. Open the docs site and open your browser's developer tools (**Network** tab).
  2. Trigger the failing action (open the assistant, run a search, or hit the MCP endpoint).
  3. Look for requests to `/_mintlify/api-public/assistant/...`, `/_mintlify/api/assistant/...`, `/mcp`, or `leaves.mintlify.com`.
  4. Note the status code and response headers:
      - `200` with no streamed body suggests a proxy is buffering server-sent events.
      - `403`, `407`, or `502` typically indicates a corporate proxy or filter is blocking the request.
      - `blocked:csp` or a CSP violation in the console means a custom [CSP header](/deploy/csp-configuration) is missing a required domain.
      - `blocked:other` usually means a browser extension or ad-blocker cancelled the request.
  5. Retry from a personal network or hotspot with all extensions disabled. If the failure disappears, the issue is on the visitor's network or browser, not on Mintlify.

  Share the request URL, status code, response headers, and any `Request ID` from the assistant error with your network administrator or with Mintlify support.
</Accordion>
