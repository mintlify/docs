<div id="introduction">
  ## Einführung
</div>

Der Web-Editor ist eine visuelle Oberfläche, mit der Sie Dokumentation direkt im Browser erstellen, bearbeiten und prüfen können.

- **Visuelles Bearbeiten**: Nehmen Sie Änderungen an Ihrer Dokumentation mit einem What-you-see-is-what-you-get-(WYSIWYG)-Editor vor, der zeigt, wie Ihre Inhalte bei der Veröffentlichung aussehen.
- **Git-Synchronisierung**: Alle Änderungen werden automatisch mit Ihrem Git-Repository synchronisiert, um die Versionskontrolle sicherzustellen.
- **Zusammenarbeit in Echtzeit**: Mehrere Teammitglieder können gleichzeitig an der Dokumentation arbeiten; Änderungen sind für alle sichtbar.
- **Keine Einrichtung erforderlich**: Beginnen Sie sofort vom Dashboard aus zu schreiben.

<div id="web-editor-flow">
  ### Ablauf im Web-Editor
</div>

So arbeiten Sie typischerweise im Web-Editor:

<Steps>
  <Step title="Wählen Sie Ihren Branch">
    Erstellen Sie einen Branch oder nehmen Sie Änderungen direkt am Deployment-Branch vor.
  </Step>

  <Step title="Öffnen Sie Ihre Datei">
    Navigieren Sie zu einer vorhandenen Datei oder erstellen Sie eine neue.
  </Step>

  <Step title="Bearbeiten Sie Ihre Inhalte">
    Nehmen Sie Änderungen im Web-Editor entweder im visuellen Modus oder im Markdown-Modus vor.
  </Step>

  <Step title="Änderungen in der Vorschau anzeigen">
    Sehen Sie, wie Ihre Änderungen im visuellen Modus erscheinen.
  </Step>

  <Step title="Änderungen veröffentlichen">
    Wenn Sie am Deployment-Branch arbeiten, veröffentlichen Sie Ihre Änderungen direkt aus dem Web-Editor. Auf anderen Branches veröffentlichen Sie Ihre Änderungen über einen Pull Request.
  </Step>
</Steps>

<div id="visual-mode">
  ### Visueller Modus
</div>

Der visuelle Modus bietet eine WYSIWYG-Oberfläche: Die Änderungen, die Sie im Editor vornehmen, sind genau die, die auf Ihrer Dokumentationsseite veröffentlicht werden. Dieser Modus ist ideal, wenn Sie Ihre Änderungen in Echtzeit sehen möchten.

<div id="markdown-mode">
  ### Markdown-Modus
</div>

Der Markdown-Modus bietet direkten Zugriff auf den zugrunde liegenden `MDX`-Code Ihrer Dokumentation. `MDX` kombiniert die Markdown-Syntax mit React-Komponenten und gibt Ihnen die volle Kontrolle über die Struktur Ihrer Inhalte. Dieser Modus ist ideal, wenn Sie Eigenschaften von Komponenten präzise steuern müssen oder wenn Sie es bevorzugen, in Markdown-Syntax zu schreiben.

<div id="git-fundamentals">
  ## Git-Grundlagen
</div>

Der Web-Editor führt im Hintergrund Git-Operationen aus. Das Verständnis dieser Konzepte hilft Ihnen, effektiver mit dem Web-Editor zu arbeiten und mit Teammitgliedern zusammenzuarbeiten, die in ihren lokalen Umgebungen arbeiten.

<AccordionGroup>
  <Accordion title="Repository">
    Die Quelle Ihrer Dokumentation, in der alle Dateien und deren Historie gespeichert sind. Der Web-Editor verbindet sich mit Ihrem Repository, um auf Inhalte zuzugreifen und sie zu bearbeiten.
  </Accordion>

  <Accordion title="Commit">
    Eine gespeicherte Momentaufnahme Ihrer Änderungen zu einem bestimmten Zeitpunkt.
  </Accordion>

  <Accordion title="Branch">
    Ein separater Arbeitsbereich, um Änderungen vorzunehmen, ohne Ihre Live-Dokumentation zu beeinträchtigen. Denken Sie daran als sichere Sandbox für Experimente und größere Updates.
  </Accordion>

  <Accordion title="Deployment branch">
    Der Haupt-Branch, der Ihre Live-Dokumentationsinhalte enthält. Änderungen an diesem Branch werden automatisch auf Ihrer Dokumentationsseite veröffentlicht. Häufig „main“ genannt.
  </Accordion>

  <Accordion title="Pull request">
    Eine Möglichkeit, das Zusammenführen Ihrer Branch-Änderungen in Ihre Live-Dokumentation vorzuschlagen. Ermöglicht Überprüfung und Diskussion, bevor Änderungen live gehen. Häufig als PR bezeichnet.
  </Accordion>

  <Accordion title="Diff">
    Ein Diff (oder Unterschied) zeigt die konkreten Änderungen zwischen zwei Versionen einer Datei. Beim Überprüfen von Pull Requests heben Diffs hervor, was hinzugefügt, entfernt oder geändert wurde.
  </Accordion>
</AccordionGroup>

<div id="what-the-web-editor-automates">
  ### Was der Web-Editor automatisiert
</div>

Der Web-Editor verbindet sich über unsere [GitHub-App](/de/settings/github) oder [GitLab-Integration](/de/settings/gitlab) mit Ihrem Git-Repository und übernimmt Git automatisch. Wenn Sie:

- **Eine Datei öffnen**: Wird die neueste Version aus Ihrem Repository abgerufen.
- **Änderungen in einer Datei vornehmen**: Werden Ihre Änderungen als Entwurf nachverfolgt, der zu einem Commit werden kann.
- **Änderungen speichern**: Wird ein Commit mit Ihren Änderungen erstellt.
- **Einen Branch erstellen**: Wird ein neuer Branch in Ihrem Repository erstellt.
- **Auf Ihrem Deployment-Branch veröffentlichen**: Wird ein Commit erstellt und direkt auf Ihren Deployment-Branch gepusht.
- **Auf anderen Branches veröffentlichen**: Wird ein Commit erstellt und ein Pull Request geöffnet.

<div id="when-to-use-branches">
  ### Wann Branches verwenden
</div>

Branches ermöglichen es Ihnen, an Änderungen zu arbeiten, ohne den Inhalt Ihrer Live-Seite zu beeinflussen. Wenn Ihre Änderungen fertig sind, können Sie sie per Pull Request in Ihren Deployment-Branch mergen.

**Best Practice: Arbeiten Sie immer in Branches.** So bleibt Ihre Live-Dokumentation stabil und ordentliche Review-Workflows werden ermöglicht.

<div id="creating-a-branch">
  ## Einen Branch erstellen
</div>

1. Wählen Sie den Branchnamen in der Editor-Toolbar aus (standardmäßig meist `main`).
2. Wählen Sie **New Branch**.
3. Geben Sie einen aussagekräftigen Namen für Ihren Branch ein, z. B. `update-getting-started` oder `fix-installation-steps`.
4. Wählen Sie **Create Branch**.

<Note>
  Möglicherweise müssen Sie Ihren neuen Branch im Dropdown-Menü auswählen, wenn der Editor nicht automatisch darauf umschaltet.
</Note>

<div id="saving-changes-on-a-branch">
  ### Änderungen auf einem Branch speichern
</div>

Um Ihre Änderungen auf einem Branch zu speichern, wählen Sie die Schaltfläche **Save Changes** oben rechts im Editor. Dadurch wird ein Commit mit Ihren Änderungen erstellt und auf Ihren Branch gepusht.

<div id="switching-branches">
  ### Zwischen Branches wechseln
</div>

1. Wählen Sie den aktuellen Branchnamen in der Editor-Toolbar aus.
2. Wählen Sie im Dropdown-Menü den Branch aus, zu dem Sie wechseln möchten.

<Tip>
  Alle nicht gespeicherten Änderungen gehen beim Wechseln des Branches verloren. Speichern oder veröffentlichen Sie Ihre Arbeit, bevor Sie wechseln.
</Tip>

<div id="making-changes">
  ## Änderungen vornehmen
</div>

Bearbeiten Sie vorhandene Inhalte, erstellen Sie neue Seiten und organisieren Sie die Struktur Ihrer Website im Web-Editor.

</div>

</div>

Drücken Sie im visuellen Modus <kbd>/</kbd>, um das Dropdown-Komponentenmenü zu öffnen. Fügen Sie Inhaltsblöcke, Hinweisboxen, Codeblöcke und andere Komponenten hinzu, um Ihre Dokumentation anzupassen.

<div id="updating-navigation">
  ### Navigation aktualisieren
</div>

Fügen Sie Seiten in Ihrer Navigation hinzu, entfernen Sie sie und ordnen Sie sie neu, indem Sie Ihre Datei `docs.json` bearbeiten:

1. Navigieren Sie im Dateiexplorer zu Ihrer `docs.json`.
2. Aktualisieren Sie die Eigenschaft `navigation`, um die gewünschte Navigationsstruktur zu erhalten. Weitere Informationen finden Sie unter [Navigation](/de/navigation).

Dieses Beispiel zeigt, wie eine Seite „Themes“ zur Gruppe „Profile“ hinzugefügt wird.

```json title="Adding a Themes page" highlight="18"
{
    "navigation": {
        "groups": [
            {
                "group": "Getting started",
                "pages": [
                    "index",
                    "quickstart",
                    "installation"
                ]
            },
            {
                "group": "Profile",
                "pages": [
                    "settings",
                    "account-types",
                    "dashboard",
                    "themes"
                ]
            }
        ]
    }
}
```

<div id="publishing-changes">
  ## Änderungen veröffentlichen
</div>

Klicke auf die Schaltfläche **Publish**, um deine Änderungen zu speichern und verfügbar zu machen.

Wie deine Änderungen veröffentlicht werden, hängt davon ab, an welchem Branch du arbeitest:

- **Deployment-Branch**: Aktualisiert deine Live-Site sofort.
- **Andere Branches**: Erstellt einen Pull Request, damit du Änderungen prüfen kannst, bevor sie live gehen.

<div id="pull-requests-and-reviewing-changes">
  ## Pull Requests und Änderungen prüfen
</div>

Pull Requests ermöglichen es Ihnen, Änderungen aus Ihrem Branch vorzuschlagen, damit andere sie vor dem Merge in Ihre Live-Dokumentation überprüfen können. So stellen Sie die Korrektheit Ihrer Änderungen sicher und geben Ihrem Team die Möglichkeit, gemeinsam am Inhalt zu arbeiten.

<Tip>
  Auch wenn Sie alleine arbeiten, sind Pull Requests hilfreich, um Änderungen vor der Veröffentlichung zu prüfen und eine klare Historie der Aktualisierungen zu behalten.
</Tip>

<div id="creating-a-pull-request-on-a-branch">
  ### Einen Pull Request aus einem Branch erstellen
</div>

Wenn Sie bereit sind, Änderungen aus Ihrem Branch zu veröffentlichen:

1. Stellen Sie sicher, dass alle Änderungen in Ihrem Branch gespeichert sind.
2. Wählen Sie **Publish Pull Request** oben rechts im Editor.
3. Fügen Sie einen Titel und eine Beschreibung für Ihren Pull Request hinzu. Ein guter Titel und eine gute Beschreibung helfen Reviewer:innen, Ihre Änderungen nachzuvollziehen.
4. Wählen Sie **Publish Pull Request**.

<Frame>
  <img src="/images/editor/pull-request-light.png" alt="Schaltfläche „Publish Pull Request“ im Mintlify-Webeditor hervorgehoben" className="block dark:hidden" />

  <img src="/images/editor/pull-request-dark.png" alt="Schaltfläche „Publish Pull Request“ im Mintlify-Webeditor hervorgehoben" className="hidden dark:block" />
</Frame>

Der Webeditor erstellt einen Pull Request in Ihrem Git-Repository und stellt einen Link zur Ansicht bereit.

<div id="reviewing-pull-requests">
  ### Pull Requests prüfen
</div>

Sobald Ihr Pull Request erstellt wurde:

1. **Änderungen prüfen**: Sie und Ihr Team können den Pull Request bei Ihrem Git-Anbieter wie GitHub oder GitLab überprüfen.
2. **Weitere Änderungen vornehmen**: Nehmen Sie nach der Überprüfung Anpassungen im Webeditor vor. Das Speichern aktualisiert den Pull Request.
3. **Bereit zum Mergen**: Wenn der Pull Request bereit ist, mergen Sie ihn, um die Änderungen auf Ihrer Live-Dokumentationsseite zu veröffentlichen.

<div id="troubleshooting">
  ## Fehlerbehebung
</div>

Hier finden Sie Lösungen für häufige Probleme mit dem Web-Editor.

<AccordionGroup>
  <Accordion title="Änderungen erscheinen nach dem Veröffentlichen nicht">
    **Mögliche Ursachen:**

    - Bereitstellung läuft noch
    - Cache-Probleme in Ihrem Browser

    **Lösungen:**

    1. Prüfen Sie den Bereitstellungsstatus in Ihrem Mintlify-Dashboard.
    2. Führen Sie einen Hard-Refresh Ihres Browsers durch (<kbd>Ctrl</kbd> + <kbd>F5</kbd> oder <kbd>Cmd</kbd> + <kbd>Shift</kbd> + <kbd>R</kbd>).
    3. Leeren Sie den Browsercache.
  </Accordion>

  <Accordion title="Berechtigungsfehler beim Veröffentlichen">
    **Mögliche Ursachen:**

    - Unzureichende Berechtigungen für das Git-Repository
    - Authentifizierungsprobleme mit Ihrem Git-Anbieter

    **Lösungen:**

    1. Stellen Sie sicher, dass Sie über den richtigen Zugriff auf das Repository verfügen.
    2. Prüfen Sie, ob Ihre Git-Integration korrekt konfiguriert ist.
    3. Lesen Sie die Dokumentation zu [Editorberechtigungen](/de/advanced/dashboard/permissions).
  </Accordion>

  <Accordion title="Probleme beim Laden des Editors">
    **Mögliche Ursachen:**

    - Netzwerkverbindungsprobleme
    - Große Dokumentations-Repositories

    **Lösungen:**

    1. Überprüfen Sie Ihre Internetverbindung.
    2. Aktualisieren Sie die Seite und versuchen Sie es erneut.
    3. Kontaktieren Sie den Support, wenn das Problem weiterhin besteht.
  </Accordion>
</AccordionGroup>
