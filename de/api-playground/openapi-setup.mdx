OpenAPI ist eine Spezifikation zur Beschreibung von APIs. Mintlify unterstützt OpenAPI‑3.0+‑Dokumente, um interaktive API-Dokumentation zu erzeugen und aktuell zu halten.

<div id="add-an-openapi-specification-file">
  ## Eine OpenAPI-Spezifikationsdatei hinzufügen
</div>

Um Ihre Endpunkte mit OpenAPI zu dokumentieren, benötigen Sie ein gültiges OpenAPI-Dokument im JSON- oder YAML-Format, das der [OpenAPI-Spezifikation 3.0+](https://swagger.io/specification/) folgt.

Sie können API-Seiten aus einem oder mehreren OpenAPI-Dokumenten erstellen.

<div id="describing-your-api">
  ### Ihre API beschreiben
</div>

Wir empfehlen die folgenden Ressourcen, um OpenAPI zu verstehen und Ihre OpenAPI-Dokumente zu erstellen.

- [Swagger OpenAPI Guide](https://swagger.io/docs/specification/v3_0/basic-structure/), um die OpenAPI-Syntax zu lernen.
- [Die Markdown-Quellen der OpenAPI-Spezifikation](https://github.com/OAI/OpenAPI-Specification/blob/main/versions/), um Details der neuesten OpenAPI-Spezifikation nachzuschlagen.
- [Swagger Editor](https://editor.swagger.io/), um Ihr OpenAPI-Dokument zu bearbeiten, zu validieren und zu debuggen.
- [Die Mint-CLI](https://www.npmjs.com/package/mint), um Ihr OpenAPI-Dokument mit dem Befehl zu validieren: `mint openapi-check <openapiFilenameOrUrl>`.

<Note>
  Der Swagger OpenAPI Guide bezieht sich auf OpenAPI v3.0, aber nahezu alle Informationen
  sind auf v3.1 anwendbar. Weitere Informationen zu den Unterschieden zwischen v3.0
  und v3.1 finden Sie im OpenAPI-Blog unter [Migration von OpenAPI 3.0 zu
  3.1.0](https://www.openapis.org/blog/2021/02/16/migrating-from-openapi-3-0-to-3-1-0).
</Note>

<div id="specifying-the-url-for-your-api">
  ### Die URL für Ihre API angeben
</div>

Um Mintlify-Funktionen wie die API-Spielwiese zu aktivieren, fügen Sie Ihrem OpenAPI-Dokument ein `servers`-Feld mit der Basis-URL Ihrer API hinzu.

```json
{
  "servers": [
    {
      "url": "https://api.example.com/v1"
    }
  ]
}
```

In einem OpenAPI-Dokument werden unterschiedliche API-Endpunkte über ihre Pfade angegeben, etwa `/users/{id}` oder einfach `/`. Die Basis-URL definiert, wo diese Pfade angehängt werden. Weitere Informationen zur Konfiguration des `servers`-Felds finden Sie in der OpenAPI-Dokumentation unter [API Server and Base Path](https://swagger.io/docs/specification/api-host-and-base-path/).

Die API-Spielwiese verwendet diese Server-URLs, um zu bestimmen, wohin Anfragen gesendet werden. Wenn Sie mehrere Server angeben, ermöglicht ein Dropdown, zwischen Servern zu wechseln. Wenn Sie keinen Server angeben, verwendet die API-Spielwiese den einfachen Modus, da ohne Basis-URL keine Anfragen gesendet werden können.

Wenn Ihre API Endpunkte hat, die unter verschiedenen URLs verfügbar sind, können Sie [das servers-Feld überschreiben](https://swagger.io/docs/specification/v3_0/api-host-and-base-path/#overriding-servers) – für einen bestimmten Pfad oder eine Operation.

<div id="specifying-authentication">
  ### Authentifizierung angeben
</div>

Um die Authentifizierung in Ihrer API-Dokumentation und in der API-Spielwiese zu aktivieren, konfigurieren Sie die Felder `securitySchemes` und `security` in Ihrem OpenAPI-Dokument. Die API-Beschreibungen und die API-Spielwiese fügen basierend auf den Sicherheitskonfigurationen in Ihrem OpenAPI-Dokument Authentifizierungsfelder hinzu.

<Steps>
  <Step title="Definieren Sie Ihre Authentifizierungsmethode.">
    Fügen Sie ein `securitySchemes`-Feld hinzu, um festzulegen, wie sich Nutzer authentifizieren.

    Dieses Beispiel zeigt eine Konfiguration für Bearer-Authentifizierung.

    ```json
    {
      "components": {
        "securitySchemes": {
          "bearerAuth": {
            "type": "http",
            "scheme": "bearer"
          }
        }
      }
    }
    ```
  </Step>

  <Step title="Wenden Sie die Authentifizierung auf Ihre Endpunkte an.">
    Fügen Sie ein `security`-Feld hinzu, um Authentifizierung zu erzwingen.

    ```json
    {
      "security": [
        {
          "bearerAuth": []
        }
      ]
    }
    ```
  </Step>
</Steps>

Zu den gängigen Authentifizierungstypen gehören:

- [API Keys](https://swagger.io/docs/specification/authentication/api-keys/): Für Header-, Query- oder Cookie-basierte Schlüssel.
- [Bearer](https://swagger.io/docs/specification/authentication/bearer-authentication/): Für JWT- oder OAuth-Tokens.
- [Basic](https://swagger.io/docs/specification/authentication/basic-authentication/): Für Benutzername und Passwort.

Wenn unterschiedliche Endpunkte innerhalb Ihrer API verschiedene Authentifizierungsmethoden erfordern, können Sie [das security-Feld überschreiben](https://swagger.io/docs/specification/authentication/#:~:text=you%20can%20apply%20them%20to%20the%20whole%20API%20or%20individual%20operations%20by%20adding%20the%20security%20section%20on%20the%20root%20level%20or%20operation%20level%2C%20respectively.) – für eine bestimmte Operation.

Weitere Informationen zum Definieren und Anwenden von Authentifizierung finden Sie in der OpenAPI-Dokumentation unter [Authentication](https://swagger.io/docs/specification/authentication/).

<div id="x-mint-extension">
  ## `x-mint`-Erweiterung
</div>

Die `x-mint`-Erweiterung ist eine benutzerdefinierte OpenAPI-Erweiterung, die zusätzliche Kontrolle darüber bietet, wie Ihre API-Dokumentation generiert und angezeigt wird.

<div id="metadata">
  ### Metadaten
</div>

Überschreiben Sie die Standardmetadaten für generierte API-Seiten, indem Sie `x-mint: metadata` zu einer beliebigen Operation hinzufügen. Sie können jedes Metadatenfeld verwenden, das im `MDX`-Frontmatter gültig wäre – mit Ausnahme von `openapi`:

```json {7-13}
{
  "paths": {
    "/users": {
      "get": {
        "summary": "Get users",
        "description": "Retrieve a list of users",
        "x-mint": {
          "metadata": {
            "title": "List all users",
            "description": "Fetch paginated user data with filtering options",
            "og:title": "Display a list of users"
          }
        },
        "parameters": [
          {
            // Parameter configuration
          }
        ]
      }
    }
  }
}
```

<div id="content">
  ### Inhalt
</div>

Fügen Sie Inhalt vor der automatisch generierten API-Dokumentation mit `x-mint: content` hinzu:

```json {6-8}
{
  "paths": {
    "/users": {
      "post": {
        "summary": "Create user",
        "x-mint": {
          "content": "## Voraussetzungen\n\nDieser Endpunkt erfordert Administratorrechte und unterliegt einem Rate-Limit.\n\n<Note>Benutzer-E-Mails müssen im gesamten System eindeutig sein.</Note>"
        },
        "parameters": [
          {
            // Parameter configuration
          }
        ]
      }
    }
  }
}
```

Die `content`-Erweiterung unterstützt alle Mintlify-MDX-Komponenten und -Formatierungen.

<div id="href">
  ### Href
</div>

Ändern Sie die URL der Endpunkt-Seite in Ihrer Dokumentation mit `x-mint: href`:

```json {6-8, 14-16}
{
  "paths": {
    "/legacy-endpoint": {
      "get": {
        "summary": "Legacy endpoint",
        "x-mint": {
          "href": "/deprecated-endpoints/legacy-endpoint"
        }
      }
    },
    "/documented-elsewhere": {
      "post": {
        "summary": "Special endpoint",
        "x-mint": {
          "href": "/guides/special-endpoint-guide"
        }
      }
    }
  }
}
```

Wenn `x-mint: href` vorhanden ist, verweist der Navigationseintrag direkt auf die angegebene URL, statt eine API-Seite zu generieren.

<div id="mcp">
  ### MCP
</div>

Stellen Sie Endpunkte selektiv als Model Context Protocol (MCP)-Tools bereit, indem Sie `x-mint: mcp` verwenden. Aktivieren Sie nur Endpunkte, die für den öffentlichen Zugriff über KI-Tools sicher sind.

<ResponseField name="mcp" type="object">
  Die MCP-Konfiguration für den Endpunkt.

  <Expandable title="MCP">
    <ResponseField name="enabled" type="boolean">
      Ob der Endpunkt als MCP-Tool bereitgestellt wird. Hat Vorrang vor der Konfiguration auf Dateiebene.
    </ResponseField>

    <ResponseField name="name" type="string">
      Der Name des MCP-Tools.
    </ResponseField>

    <ResponseField name="description" type="string">
      Die Beschreibung des MCP-Tools.
    </ResponseField>
  </Expandable>
</ResponseField>

<CodeGroup>
  ```json Selective enablement {6-9} wrap
  {
    "paths": {
      "/users": {
        "post": {
          "summary": "Create user",
          "x-mint": {
            "mcp": {
              "enabled": true
            },
            // ...
          }
        }
      },
      "/users": {
        "delete": {
          "summary": "Delete user (admin only)",
          // No `x-mint: mcp` so this endpoint is not exposed as an MCP tool
          // ...
        }
      }
    }
  }
  ```

  ```json Global enablement {3-5, 9-13} wrap
  {
    "openapi": "3.1.0",
    "x-mcp": {
        "enabled": true // All endpoints are exposed as MCP tools by default
      },
    "paths": {
      "/api/admin/delete": {
        "delete": {
          "x-mint": {
            "mcp": {
              "enabled": false // Disable MCP for this endpoint
            }
          },
          "summary": "Delete resources"
        }
      }
    }
  }
  ```
</CodeGroup>

Weitere Informationen finden Sie unter [Model Context Protocol](/de/ai/model-context-protocol).

<div id="auto-populate-api-pages">
  ## API-Seiten automatisch befüllen
</div>

Fügen Sie einem Navigationselement in Ihrer `docs.json` ein `openapi`-Feld hinzu, um Seiten für OpenAPI-Endpunkte automatisch zu generieren. Sie können steuern, wo diese Seiten in Ihrer Navigationsstruktur erscheinen – als eigene API-Bereiche oder zusammen mit anderen Seiten.

Das `openapi`-Feld akzeptiert entweder einen Dateipfad in Ihrem Docs-Repository oder eine URL zu einem gehosteten OpenAPI-Dokument.

Generierte Endpunktseiten haben folgende Standardmetadaten:

- `title`: Das `summary`-Feld der Operation, falls vorhanden. Ist kein `summary` vorhanden, wird der Titel aus der HTTP-Methode und dem Endpunkt generiert.
- `description`: Das `description`-Feld der Operation, falls vorhanden.
- `version`: Der `version`-Wert vom übergeordneten Anker oder Tab, falls vorhanden.
- `deprecated`: Das `deprecated`-Feld der Operation. Wenn `true`, erscheint ein „Veraltet“-Label neben dem Endpunkttitel in der Seitennavigation und auf der Endpunktseite.

<Tip>
  Um bestimmte Endpunkte von Ihren automatisch generierten API-Seiten auszuschließen, fügen Sie der
  [x-hidden](/de/api-playground/customization/managing-page-visibility#x-hidden)
  Eigenschaft die Operation in Ihrer OpenAPI-Spezifikation hinzu.
</Tip>

Es gibt zwei Ansätze, um Endpunktseiten in Ihre Dokumentation aufzunehmen:

1. Dedizierte API-Bereiche: Referenzieren Sie OpenAPI-Spezifikationen in Navigationselementen für dedizierte API-Bereiche.
2. Selektive Endpunkte: Referenzieren Sie bestimmte Endpunkte in Ihrer Navigation neben anderen Seiten.

<div id="dedicated-api-sections">
  ### Dedizierte API-Bereiche
</div>

Generieren Sie dedizierte API-Bereiche, indem Sie einem Navigationselement ein `openapi`-Feld hinzufügen und keine weiteren Seiten. Alle Endpunkte in der Spezifikation werden aufgenommen:

```json {5}
"navigation": {
  "tabs": [
    {
        "tab": "API Reference",
        "openapi": "https://petstore3.swagger.io/api/v3/openapi.json"
    }
  ]
}
```

Sie können mehrere OpenAPI-Spezifikationen in verschiedenen Navigationsbereichen verwenden:

```json {8-11, 15-18}
"navigation": {
  "tabs": [
    {
      "tab": "API Reference",
      "groups": [
        {
          "group": "Users",
          "openapi": {
            "source": "/path/to/openapi-1.json",
            "directory": "api-reference"
          }
        },
        {
          "group": "Admin",
          "openapi": {
            "source": "/path/to/openapi-2.json",
            "directory": "api-reference"
          }
        }
      ]
    }
  ]
}
```

<Note>
  Das `directory`-Feld ist optional und gibt an, wo generierte API-Seiten
  in Ihrem Docs-Repository gespeichert werden. Wenn nicht angegeben, wird standardmäßig das
  Verzeichnis `api-reference` Ihres Repositories verwendet.
</Note>

<div id="selective-endpoints">
  ### Selektive Endpunkte
</div>

Wenn Sie mehr Kontrolle darüber wünschen, wo Endpunkte in Ihrer Dokumentation erscheinen, können Sie bestimmte Endpunkte in Ihrer Navigation referenzieren. Dieser Ansatz ermöglicht es Ihnen, Seiten für API-Endpunkte neben anderen Inhalten zu generieren.

<div id="set-a-default-openapi-spec">
  #### Eine Standard-OpenAPI-Spezifikation festlegen
</div>

Konfigurieren Sie eine Standard-OpenAPI-Spezifikation für ein Navigationselement. Referenzieren Sie dann bestimmte Endpunkte im `pages`-Feld:

```json {12, 15-16}
"navigation": {
  "tabs": [
    {
      "tab": "Getting started",
      "pages": [
        "quickstart",
        "installation"
      ]
    },
    {
      "tab": "API reference",
      "openapi": "/path/to/openapi.json",
      "pages": [
        "api-overview",
        "GET /users",
        "POST /users",
        "guides/authentication"
      ]
    }
  ]
}
```

Jeder Seiteneintrag, der dem Format `METHOD /path` entspricht, erzeugt eine API-Seite für diesen Endpunkt unter Verwendung der Standard-OpenAPI-Spezifikation.

<div id="openapi-spec-inheritance">
  #### Vererbung der OpenAPI-Spezifikation
</div>

OpenAPI-Spezifikationen werden entlang der Navigationshierarchie vererbt. Untergeordnete Navigationselemente erben die OpenAPI-Spezifikation ihres übergeordneten Elements, sofern sie nicht ihre eigene definieren:

```json {3, 7-8, 11, 13-14}
{
  "group": "API reference",
  "openapi": "/path/to/openapi-v1.json",
  "pages": [
    "overview",
    "authentication",
    "GET /users",
    "POST /users",
    {
      "group": "Orders",
      "openapi": "/path/to/openapi-v2.json",
      "pages": [
        "GET /orders",
        "POST /orders"
      ]
    }
  ]
}
```

<div id="individual-endpoints">
  #### Individuelle Endpunkte
</div>

Referenzieren Sie bestimmte Endpunkte, ohne eine Standard-OpenAPI-Spezifikation festzulegen, indem Sie den Dateipfad angeben:

```json {5-6}
"navigation": {
  "pages": [
    "introduction",
    "user-guides",
    "/path/to/openapi-v1.json POST /users",
    "/path/to/openapi-v2.json GET /orders"
  ]
}
```

Dieser Ansatz ist hilfreich, wenn Sie einzelne Endpunkte aus verschiedenen Spezifikationen benötigen oder nur ausgewählte Endpunkte einbinden möchten.

<div id="create-mdx-files-for-api-pages">
  ## `MDX`-Dateien für API-Seiten erstellen
</div>

Um die Kontrolle über einzelne Endpunktseiten zu behalten, erstellen Sie für jede Operation `MDX`-Seiten. So können Sie Seitenmetadaten anpassen, Inhalte hinzufügen, bestimmte Operationen auslassen oder Seiten in Ihrer Navigation auf Seitenebene neu anordnen.

Sehen Sie sich ein [Beispiel für eine MDX-OpenAPI-Seite von MindsDB](https://github.com/mindsdb/mindsdb/blob/main/docs/rest/databases/create-databases.mdx?plain=1) an und wie sie in deren [Live-Dokumentation](https://docs.mindsdb.com/rest/databases/create-databases) erscheint.

<div id="manually-specify-files">
  ### Dateien manuell angeben
</div>

Erstellen Sie für jeden Endpunkt eine `MDX`-Seite und geben Sie mithilfe des Felds `openapi` im Frontmatter an, welche OpenAPI-Operation angezeigt werden soll.

Wenn Sie auf diese Weise eine OpenAPI-Operation referenzieren, werden Name, Beschreibung, Parameter, Antworten und die API-Spielwiese automatisch aus Ihrem OpenAPI-Dokument generiert.

Wenn Sie mehrere OpenAPI-Dateien haben, fügen Sie den Dateipfad in Ihre Referenz ein, damit Mintlify das richtige OpenAPI-Dokument findet. Wenn Sie nur eine OpenAPI-Datei haben, erkennt Mintlify diese automatisch.

<Note>
  Dieser Ansatz funktioniert unabhängig davon, ob Sie eine Standard-OpenAPI-Spezifikation
  in Ihrer Navigation festgelegt haben. Sie können jeden Endpunkt aus jeder OpenAPI-Spezifikation referenzieren, indem Sie
  den Dateipfad im Frontmatter angeben.
</Note>

Wenn Sie eine externe OpenAPI-Datei referenzieren möchten, fügen Sie die URL der Datei zu Ihrer `docs.json` hinzu.

<CodeGroup>
  ```mdx Example
  ---
  title: "Get users"
  description: "Returns all plants from the system that the user has access to"
  openapi: "/path/to/openapi-1.json GET /users"
  deprecated: true
  version: "1.0"
  ---
  ```

  ```mdx Format
  ---
  title: "title of the page"
  description: "description of the page"
  openapi: openapi-file-path method path
  deprecated: boolean (not required)
  version: "version-string" (not required)
  ---
  ```
</CodeGroup>

<Note>
  Methode und Pfad müssen exakt der Definition in Ihrer OpenAPI-
  Spezifikation entsprechen. Wenn der Endpunkt in der OpenAPI-Datei nicht existiert, bleibt die Seite
  leer.
</Note>

<div id="autogenerate-mdx-files">
  ### `MDX`-Dateien automatisch generieren
</div>

Verwenden Sie unseren Mintlify-[Scraper](https://www.npmjs.com/package/@mintlify/scraping), um `MDX`-Seiten für umfangreiche OpenAPI-Dokumente automatisch zu generieren.

<Note>
  Ihr OpenAPI-Dokument muss gültig sein, sonst werden die Dateien nicht automatisch generiert.
</Note>

Der Scraper generiert:

- Eine `MDX`-Seite für jede Operation im Feld `paths` Ihres OpenAPI-Dokuments.
- Wenn Ihr OpenAPI-Dokument Version 3.1+ ist, eine `MDX`-Seite für jede Operation im Feld `webhooks` Ihres OpenAPI-Dokuments.
- Ein Array von Navigationseinträgen, das Sie zu Ihrer `docs.json` hinzufügen können.

<Steps>
  <Step title="`MDX`-Dateien generieren.">
    ```bash
    npx @mintlify/scraping@latest openapi-file <path-to-openapi-file>
    ```
  </Step>

  <Step title="Einen Ausgabefolder angeben.">
    ```bash
    npx @mintlify/scraping@latest openapi-file <path-to-openapi-file> -o api-reference
    ```

    Fügen Sie das Flag `-o` hinzu, um einen Ordner anzugeben, in den die Dateien geschrieben werden. Wenn kein Ordner angegeben ist, werden die Dateien im Arbeitsverzeichnis erstellt.
  </Step>
</Steps>

<div id="create-mdx-files-for-openapi-schemas">
  ### `MDX`-Dateien für OpenAPI-Schemas erstellen
</div>

Sie können individuelle Seiten für jedes OpenAPI-Schema erstellen, das im Feld `components.schema` eines OpenAPI-Dokuments definiert ist:

<CodeGroup>
  ```mdx Example
  ---
  openapi-schema: OrderItem
  ---
  ```

  ```mdx Format
  ---
  openapi-schema: "schema-key"
  ---
  ```
</CodeGroup>

<div id="webhooks">
  ## Webhooks
</div>

Webhooks sind HTTP-Callbacks, die Ihre API sendet, um externe Systeme bei Ereignissen zu benachrichtigen. Webhooks werden in OpenAPI-3.1+-Dokumenten unterstützt.

<div id="define-webhooks-in-your-openapi-specification">
  ### Webhooks in Ihrer OpenAPI-Spezifikation definieren
</div>

Fügen Sie Ihrem OpenAPI-Dokument neben dem Feld `paths` das Feld `webhooks` hinzu.

Weitere Informationen zur Definition von Webhooks finden Sie unter [Webhooks](https://spec.openapis.org/oas/v3.1.0#oasWebhooks) in der OpenAPI-Dokumentation.

<div id="reference-webhooks-in-mdx-files">
  ### Webhooks in MDX-Dateien referenzieren
</div>

Wenn Sie MDX-Seiten für Webhooks erstellen, verwenden Sie `webhook` statt HTTP-Methoden wie `GET` oder `POST`:

```mdx
---
title: "Example webhook"
description: "Triggered when an event occurs"
openapi: "path/to/openapi-file webhook example-webhook-name"
---
```

<Note>
  Der Webhook-Name muss exakt dem Schlüssel entsprechen, der im Feld `webhooks` Ihrer OpenAPI-Spezifikation definiert ist.
</Note>
