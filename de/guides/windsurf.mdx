Mache Windsurf mit Arbeitsbereichsregeln und Memories zu einem Dokumentationsprofi, der deinen Styleguide, die Komponenten und den Projektkontext versteht.

<div id="using-windsurf-with-mintlify">
  ## Verwendung von Windsurf mit Mintlify
</div>

Der Cascade-AI-Assistent von Windsurf lässt sich so konfigurieren, dass er mit Mintlify-Komponenten Dokumentation nach Ihren Standards erstellt. Workspace-Regeln und Memories liefern dauerhaften Kontext zu Ihrem Projekt und sorgen für konsistentere Vorschläge von Cascade.

* **Workspace-Regeln** werden in Ihrem Dokumentations-Repository gespeichert und mit Ihrem Team geteilt.
* **Memories** bieten individuellen Kontext, der sich im Laufe der Zeit aufbaut.

Wir empfehlen, Workspace-Regeln für gemeinsame Dokumentationsstandards einzurichten. Sie können Memories während der Arbeit aufbauen; da sie jedoch nicht geteilt werden, sind sie nicht über alle Teammitglieder hinweg einheitlich.

Erstellen Sie Workspace-Regeln im Verzeichnis `.windsurf/rules` Ihres Docs-Repos. Weitere Informationen finden Sie unter [Memories & Rules](https://docs.windsurf.com/windsurf/cascade/memories) in der Windsurf-Dokumentation.

<div id="example-workspace-rule">
  ## Beispiel-Regel für den Arbeitsbereich
</div>

Diese Regel liefert Cascade Kontext zu Mintlify-Komponenten und allgemeinen Best Practices für technische Dokumentation.

Sie können diese Beispielregel unverändert verwenden oder für Ihre Dokumentation anpassen:

* **Schreibstandards**: Aktualisieren Sie die Sprachrichtlinien, damit sie Ihrem Styleguide entsprechen.
* **Komponentenmuster**: Fügen Sie projektspezifische Komponenten hinzu oder passen Sie bestehende Beispiele an.
* **Codebeispiele**: Ersetzen Sie generische Beispiele durch echte API-Aufrufe und -Antworten für Ihr Produkt.
* **Stil- und Tonpräferenzen**: Passen Sie Terminologie, Formatierung und andere Regeln an.

Speichern Sie Ihre Regel als `.md`-Datei im Verzeichnis `.windsurf/rules` Ihres Docs-Repos.

````mdx
# Mintlify technical writing rule

## Project context

- This is a documentation project on the Mintlify platform
- We use MDX files with YAML frontmatter  
- Navigation is configured in `docs.json`
- We follow technical writing best practices

## Writing standards

- Use second person ("you") for instructions
- Write in active voice and present tense
- Start procedures with prerequisites
- Include expected outcomes for major steps
- Use descriptive, keyword-rich headings
- Keep sentences concise but informative

## Required page structure

Every page must start with frontmatter:

```yaml
---
title: "Clear, specific title"
description: "Concise description for SEO and navigation"
---
```

## Mintlify components

### docs.json

- Refer to the [docs.json schema](https://mintlify.com/docs.json) when building the docs.json file and site navigation

### Callouts

- `<Note>` for helpful supplementary information
- `<Warning>` for important cautions and breaking changes
- `<Tip>` for best practices and expert advice  
- `<Info>` for neutral contextual information
- `<Check>` for success confirmations

### Code examples

- When appropriate, include complete, runnable examples
- Use `<CodeGroup>` for multiple language examples
- Specify language tags on all code blocks
- Include realistic data, not placeholders
- Use `<RequestExample>` and `<ResponseExample>` for API docs

### Procedures

- Use `<Steps>` component for sequential instructions
- Include verification steps with `<Check>` components when relevant
- Break complex procedures into smaller steps

### Content organization

- Use `<Tabs>` for platform-specific content
- Use `<Accordion>` for progressive disclosure
- Use `<Card>` and `<CardGroup>` for highlighting content
- Wrap images in `<Frame>` components with descriptive alt text

## API documentation requirements

- Document all parameters with `<ParamField>` 
- Show response structure with `<ResponseField>`
- Include both success and error examples
- Use `<Expandable>` for nested object properties
- Always include authentication examples

## Quality standards

- Test all code examples before publishing
- Use relative paths for internal links
- Include alt text for all images
- Ensure proper heading hierarchy (start with h2)
- Check existing patterns for consistency
````

<div id="working-with-cascade">
  ## Arbeiten mit Cascade
</div>

Sobald Ihre Regeln eingerichtet sind, können Sie Cascade für verschiedene Dokumentationsaufgaben einsetzen. Weitere Informationen finden Sie in der Windsurf-Dokumentation unter [Cascade](https://docs.windsurf.com/windsurf/cascade).

<div id="example-prompts">
  ### Beispielanweisungen
</div>

**Neue Inhalte verfassen**:
```text wrap
Erstellen Sie eine neue Seite, die erklärt, wie sich Nutzer bei unserer API authentifizieren. Fügen Sie Codebeispiele in JavaScript, Python und cURL hinzu.
```

**Bestehende Inhalte verbessern**:
```text wrap
Überarbeiten Sie diese Seite und schlagen Sie Verbesserungen in Bezug auf Verständlichkeit und den Einsatz von Komponenten vor. Konzentrieren Sie sich darauf, die Schritte leichter nachvollziehbar zu machen.
```

**Codebeispiele erstellen**:
```text wrap
Erzeugen Sie ein vollständiges Codebeispiel, das die Fehlerbehandlung für diesen API-Endpunkt demonstriert. Verwenden Sie realistische Daten und fügen Sie erwartete Antworten hinzu.
```

**Konsistenz wahren**:
```text wrap
Prüfen Sie, ob diese neue Seite unseren Dokumentationsstandards entspricht, und schlagen Sie gegebenenfalls notwendige Änderungen vor.
```
