Git 是一种版本控制系统，可让你跟踪文档更改并与团队成员协作。借助 Git，你可以查看项目中每个文件在何时、发生了哪些更改，以及更改的原因。必要时，你也可以轻松将文件恢复到先前版本。

Web 编辑器在后台会执行一些 Git 操作。理解这些概念将帮助你更高效地使用 Web 编辑器，并与在本地开发环境中工作的团队成员更好地协作。

<div id="core-git-concepts">
  ## 核心 Git 概念
</div>

<AccordionGroup>
  <Accordion title="Repository">
    你的文档源码所在之处，保存所有文件及其历史记录。Web 编辑器会连接到你的存储库以访问并修改内容。
  </Accordion>

  <Accordion title="Commit">
    在某个时间点保存的更改快照。每次提交都会包含描述变更内容的消息，并在项目历史中创建永久记录。
  </Accordion>

  <Accordion title="Branch">
    一个 branch 指向你存储库中的特定提交。你的在线文档是从部署用分支构建的。你可以拥有任意数量的其他分支，这些分支上的更改尚未发布到在线文档。如果你想将某个分支的更改纳入在线文档，可以通过拉取请求将该分支合并到部署用分支中。

    使用分支可以在不影响在线文档的情况下开展工作，安全试验新功能，并在发布前获取评审。
  </Accordion>

  <Accordion title="Deployment branch">
    项目的主要分支，在线文档内容由此构建。对该分支的更改会自动发布到你的文档站点。常见名称为 `main`，但你可以将任何分支设置为部署用分支。
  </Accordion>

  <Accordion title="Pull request">
    一种提议将某个分支上的更改合并进在线文档的方式。允许在更改上线前进行评审和讨论。通常称为 PR，在 GitLab 中也称为合并请求（merge request）。
  </Accordion>

  <Accordion title="Diff">
    diff（差异）展示同一文件两个版本之间的更改。在审查拉取请求时，diff 会突出显示新增、删除或修改的内容，便于识别变更。
  </Accordion>
</AccordionGroup>

<div id="how-the-web-editor-uses-git">
  ## Web 编辑器如何使用 Git
</div>

Web 编辑器通过 [GitHub 应用](/zh/settings/github) 或 [GitLab 集成](/zh/settings/gitlab) 连接到你的 Git 存储库，并自动化常见的 Git 操作。

当你：

* **打开文件**：编辑器会从你的存储库获取最新版本，确保你始终处理的是最新内容。
* **进行更改**：编辑器会将你的更改作为草稿进行跟踪；当你准备好保存时，这些更改将变为一次提交。
* **保存更改**：编辑器会将你的更改提交，保存在项目历史中。
* **创建 branch**：编辑器会在你的存储库中创建一个新的 branch，具有存储库访问权限的任何人都可以使用它进行协作和审阅更改。
* **在你的部署用分支上发布**：编辑器会直接向你的部署用分支提交并推送，更改会立即发布。
* **在其他分支上发布**：编辑器会创建一个拉取请求（PR；亦称“合并请求”/Merge Request），让你在将更改合并到部署用分支之前先获得他人的反馈。

<div id="git-best-practices">
  ## Git 最佳实践
</div>

每个团队都会形成自己的工作流和偏好，但下面是一些通用的最佳实践，助你快速上手。

* **撰写清晰的提交消息**：使用主动语态，具体说明改动内容。
* **使用具描述性的 branch 名称**：branch 名称应能说明正在进行的工作，并让查看你存储库分支的人一目了然。
* **保持分支聚焦**：让某个 branch 上的改动专注于一个特定任务或项目。
* **合并后删除分支**：不再需要时删除 branch，以保持存储库整洁。
