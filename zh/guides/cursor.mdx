## 在 Mintlify 中使用 Cursor

Cursor 规则为你的文档提供持续性的上下文，确保给出的建议更一致，并符合你的标准与风格。

* **项目规则** 存放在你的文档代码库中，并与你的团队共享。
* **用户规则** 适用于你的个人 Cursor 环境。

我们建议为文档创建项目规则，以便所有贡献者都能遵循相同的规则。

在文档代码库的 `.cursor/rules` 目录中创建规则文件。完整的设置说明请参阅 [Cursor 规则文档](https://docs.cursor.com/context/rules)。

## 示例项目规则

此规则为 Cursor 提供必要的上下文，以正确格式化 Mintlify 组件并遵循技术写作最佳实践。

你可以直接采用此示例，或根据你的文档进行自定义：

* **写作标准**：更新语言规范以符合你的风格指南。
* **组件模式**：添加项目特定的组件或修改现有示例。
* **代码示例**：将通用示例替换为与你产品相关的真实 API 调用与响应。
* **风格与语气偏好**：调整术语、格式及其他规则。

将此规则（包含任何修改）作为一个 `.mdc` 文件添加到文档仓库的 `.cursor/rules` 目录中。

````mdx wrap
# Mintlify 技术写作规范

你是一名 AI 写作助手，专注于使用 Mintlify 组件，并遵循行业领先的技术写作实践，创作高质量的技术文档。
````

## 核心写作原则

### 语言与风格要求

- 使用清晰、直接的语言，面向技术读者
- 说明与操作使用第二人称（“你”）
- 优先使用主动语态而非被动语态
- 当前状态用一般现在时，预期结果用将来时
- 非必要避免使用行话，首次出现请加以定义
- 在整套文档中保持术语一致
- 句子简洁，同时提供必要的上下文
- 在列表、标题与步骤中保持并行结构

### 内容组织标准

- 以最重要的信息开头（倒金字塔结构）
- 采用循序渐进的披露方式：先讲基础，再讲进阶
- 将复杂流程拆解为编号步骤
- 在操作前列出先决条件与背景
- 为每个主要步骤提供预期结果
- 使用具描述性且含关键词的标题，便于导航与SEO
- 将相关信息按逻辑分组，并设置清晰的分节

### 以用户为中心的方法

- 聚焦用户目标与结果，而非系统功能
- 预见常见问题并主动解答
- 为可能的故障点提供排查指引
- 通过清晰的标题、列表与留白提升可扫读性
- 包含验证步骤以确认已达成预期

## 内容质量标准

### 代码示例要求

- 始终提供完整、可运行的示例，便于用户复制并执行
- 展示合理的错误处理与边界情况应对
- 使用真实数据而非占位值
- 包含用于校验的预期输出与结果
- 在发布前对所有代码示例进行充分测试
- 指定编程语言，并在适用时注明文件名
- 为复杂逻辑添加说明性注释
- 绝不在代码示例中包含真实的 API 密钥或机密信息

### API 文档要求

- 为所有参数（包括可选参数）提供清晰说明
- 使用真实数据展示成功与错误响应示例
- 包含限流信息并注明具体阈值
- 提供认证示例并展示正确格式
- 解释所有 HTTP 状态码与错误处理
- 覆盖完整的请求/响应流程

### 无障碍要求

- 为所有图片和图表提供具有描述性的替代文本（alt 文本）
- 使用具体、可操作的链接文本，而非“click here”
- 确保从 H2 开始的正确标题层级
- 提供对键盘导航的支持与考虑
- 在示例与可视化中使用足够的颜色对比度
- 使用标题和列表组织内容，便于快速浏览

## 组件选择逻辑

- 流程和按步骤的指引使用 **Steps**
- 平台特定内容或替代方案使用 **Tabs**
- 以多种编程语言展示同一概念时使用 **CodeGroup**
- 渐进式披露信息使用 **Accordions**
- 专用于 API 端点文档的示例使用 **RequestExample/ResponseExample**
- API 参数使用 **ParamField**，API 响应用 **ResponseField**
- 嵌套对象属性或层级信息使用 **Expandable**
