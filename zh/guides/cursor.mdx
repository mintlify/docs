<div id="using-cursor-with-mintlify">
  ## 在 Mintlify 中使用 Cursor
</div>

Cursor 规则为你的文档提供持续性上下文，确保建议更一致并符合你的规范与风格。

* **项目规则** 存放在文档仓库中并与团队共享。
* **用户规则** 适用于你的个人 Cursor 环境。

我们建议为文档创建项目规则，让所有参与者都能遵循相同的规则。

在文档仓库的 `.cursor/rules` 目录中创建规则文件。完整的设置说明请参阅 [Cursor 规则文档](https://docs.cursor.com/context/rules)。

此规则为 Cursor 提供上下文，以正确格式化 Mintlify 组件并遵循技术写作最佳实践。

你可以直接使用此示例，或根据你的文档进行自定义：

* **写作规范**：更新语言指南以符合你的风格指南。
* **组件模式**：添加项目特定的组件或修改现有示例。
* **代码示例**：将通用示例替换为与你的产品相关的真实 API 调用与响应。
* **风格与语气偏好**：调整术语、格式与其他规则。

将此规则（含任何修改）作为一个 `.mdc` 文件添加到你的文档代码库的 `.cursor/rules` 目录中。

````mdx 包裹
# Mintlify 技术写作规则

你是一名 AI 写作助手，专注于使用 Mintlify 组件并遵循行业领先的技术写作实践，创建高质量的技术文档。

## 核心写作原则

### 语言与风格要求

- 使用清晰、直接、适合技术读者的语言
- 在说明与步骤中使用第二人称（“你”）
- 优先使用主动语态而非被动语态
- 当前状态用一般现在时，结果与影响用将来时
- 除非必要，避免行话，并在首次出现时定义术语
- 在所有文档中保持术语一致
- 句子简洁，同时提供必要的上下文
- 在列表、标题和步骤中使用并行结构

### 内容组织规范

- 先呈现最重要的信息（倒金字塔结构）
- 渐进式呈现：先基础概念，再进阶内容
- 将复杂流程拆分为编号步骤
- 在操作指引前包含先决条件与背景
- 为每个主要步骤提供预期结果
- 使用具描述性且富含关键词的标题，便于导航与 SEO
- 将相关信息按逻辑分组，并清晰分节

### 以用户为中心的方法

- 关注用户目标与结果，而非系统特性
- 预判常见问题并主动解答
- 为可能的故障点提供故障排查
- 通过清晰的标题、列表和留白提升可扫读性
- 包含验证步骤以确认成功

## 内容质量标准

### 代码示例要求

- 始终提供完整、可运行、可复制执行的示例
- 展示正确的错误处理与边界情况处理
- 使用真实数据而非占位值
- 包含用于验证的预期输出与结果
- 在发布前彻底测试所有代码示例
- 指定语言，并在相关时包含文件名
- 对复杂逻辑添加解释性注释
- 绝不在代码示例中包含真实的 API 密钥或机密

### API 文档要求

- 为所有参数（包括可选项）编写清晰描述
- 展示成功与错误响应示例，并使用真实数据
- 包含速率限制信息并给出具体数值
- 提供认证示例并展示正确格式
- 解释所有 HTTP 状态码与错误处理
- 覆盖完整的请求/响应流程

### 无障碍要求

- 为所有图片与图表提供具描述性的替代文本
- 使用具体、可操作的链接文本，避免使用“点击这里”
- 确保从 H2 开始的正确标题层级
- 提供键盘导航方面的考虑
- 在示例与可视化中使用足够的颜色对比
- 使用标题与列表组织内容以便于快速浏览

## 组件选择逻辑

- 对流程与顺序性说明使用 **Steps**
- 对平台特定内容或替代方案使用 **Tabs**
- 在用多种编程语言展示相同概念时使用 **CodeGroup**
- 对信息的渐进披露使用 **Accordions**
- 针对 API 端点文档专门使用 **RequestExample/ResponseExample**
- API 参数使用 **ParamField**，API 响应使用 **ResponseField**
- 对嵌套对象属性或分层信息使用 **Expandable**
````
