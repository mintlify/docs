可从以下三种方式中选择，根据用户信息与安全需求来管理访问并自定义文档：

- **完全认证**：为全部内容提供完整的隐私保护，并支持全面的内容定制。
- **部分认证**：按页面进行访问控制，并支持全面的内容定制。
- **个性化**：支持内容定制，但**不提供任何安全保证**。所有内容仍对公众可见。

### 使用场景示例

**在以下情况下选择完全认证：**

- 构建包含敏感信息的公司内部文档
- 记录需要用户验证的专有 API
- 创建面向特定客户的实施指南

**在以下情况下选择部分认证：**

- 提供公开的入门指南，同时将高级功能文档设为私有
- 运营免费增值产品，让付费用户获得额外文档
- 发布开源文档，并包含私有的企业版部分

**在以下情况下选择个性化：**

- 创建公开的 API 文档，并展示用户特定的示例
- 构建可根据用户画像自定义内容的营销站点
- 提供可根据用户偏好或技能水平自适应的公开教程

## 握手方式

选择最符合你现有基础设施与安全要求的方式。

| 方法 | 适用范围 | 设置复杂度 | 最佳适用场景 |
|:-------|:--------------|:-----------------|:----------|
| **JWT** | 所有方案 | 中等 | 自定义登录流程，最大安全可控性 |
| **OAuth 2.0** | 所有方案 | 高 | 第三方认证提供商，企业级安全 |
| **Mintlify Dashboard** | 仅认证 | 低 | 已使用 Mintlify 控制台的团队 |
| **Password** | 仅认证 | 低 | 无需个性化的简单共享访问 |
| **Shared Session** | 仅个性化 | 低 | 已有会话式认证的应用 |

### 何时使用各方式

**JWT**：当你已有认证系统并希望完全掌控登录流程时使用。适用于自定义用户管理，或需要将文档访问与主应用解耦的场景。

**OAuth 2.0**：当你想利用第三方认证提供商（Google、GitHub 等）或需要企业级安全标准时使用。最适合已在使用 OAuth 基础设施的组织。

**Mintlify Dashboard**：当你的文档编辑者同时也是文档读者时使用。非常适合已通过 Mintlify 控制台管理内容的内部团队。

**Password**：当你不需要跟踪个体用户或个性化内容、只需简单访问控制时使用。适用于外包/合作方、内测用户或临时访问场景。

**Shared Session**：当你希望在应用与文档之间实现无缝登录时使用。适用于用户已在主应用中完成认证，并希望为其文档体验实现个性化的场景。

## 内容自定义

这三种方式都支持内容个性化能力，包括动态 MDX 内容、API 密钥预填、以及页面可见性控制。有关详细的实现指南，请参见[个性化设置](/zh/authentication-personalization/personalization-setup)。
