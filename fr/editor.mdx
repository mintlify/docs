## Introduction

L’Éditeur web est une interface visuelle pour créer, modifier et relire la documentation directement dans votre navigateur.

- **Édition visuelle** : Modifiez votre documentation avec un éditeur WYSIWYG (ce que vous voyez est ce que vous obtenez) qui affiche le rendu final de votre contenu.
- **Synchronisation Git** : Toutes les modifications se synchronisent automatiquement avec votre dépôt Git pour assurer le versionnage.
- **Collaboration en temps réel** : Plusieurs membres de l’équipe peuvent travailler simultanément sur la documentation, avec des modifications visibles par tous.
- **Aucune configuration requise** : Commencez à rédiger immédiatement depuis votre tableau de bord.

### Parcours dans l’Éditeur web

Voici comment vous travaillerez généralement dans l’Éditeur web :

<Steps>
  <Step title="Choose your branch">
    Créez une branche ou effectuez des modifications directement sur votre branche de déploiement.
  </Step>
  <Step title="Open your file">
    Accédez à un fichier existant ou créez-en un nouveau.
  </Step>
  <Step title="Edit your content">
    Modifiez votre contenu dans l’Éditeur web en mode visuel ou en mode Markdown.
  </Step>
  <Step title="Preview your changes">
    Prévisualisez vos modifications en mode visuel.
  </Step>
  <Step title="Publish your changes">
    Si vous travaillez sur votre branche de déploiement, publiez vos modifications directement depuis l’Éditeur web. Sur les autres branches, publiez vos modifications via une pull request.
  </Step>
</Steps>

### Mode visuel

Le mode visuel offre une expérience WYSIWYG : les modifications que vous effectuez dans l’éditeur sont celles qui seront publiées sur votre site de documentation. Ce mode est idéal lorsque vous souhaitez voir en temps réel le rendu de vos changements.

### Mode Markdown

Le mode Markdown donne un accès direct au code MDX (Markdown avec composants React) de votre documentation. MDX combine la syntaxe Markdown avec des composants React, vous offrant un contrôle total sur la structure de votre contenu. Ce mode est idéal lorsque vous avez besoin d’un contrôle précis sur les propriétés des composants ou si vous préférez écrire en syntaxe Markdown.

## Principes fondamentaux de Git

L’éditeur web effectue des opérations Git en arrière-plan. Comprendre ces concepts vous aidera à travailler plus efficacement avec l’éditeur web et à collaborer avec les membres de l’équipe qui travaillent en local.

<AccordionGroup>
  <Accordion title="Repository">
    La source de votre documentation où tous les fichiers et leur historique sont stockés. L’éditeur web se connecte à votre repository pour accéder au contenu et le modifier.
  </Accordion>

  <Accordion title="Commit">
    Un instantané enregistré de vos modifications à un moment donné.
  </Accordion>

  <Accordion title="Branch">
    Un espace de travail distinct pour effectuer des modifications sans affecter votre documentation en production. Voyez cela comme un bac à sable sécurisé pour des expérimentations et des mises à jour plus importantes.
  </Accordion>

  <Accordion title="Deployment branch">
    La branche principale qui contient le contenu de votre documentation en ligne. Les modifications apportées à cette branche sont automatiquement publiées sur votre site de documentation. Souvent appelée « main ».
  </Accordion>

  <Accordion title="Pull request">
    Un moyen de proposer la fusion des modifications de votre branche dans votre documentation en ligne. Permet la relecture et la discussion avant la mise en production. Couramment appelée PR.
  </Accordion>

  <Accordion title="Diff">
    Un diff (ou différence) montre les changements spécifiques entre deux versions d’un fichier. Lors de la revue des pull requests, les diffs mettent en évidence ce qui a été ajouté, supprimé ou modifié.
</Accordion>
</AccordionGroup>

### Ce que l’éditeur web automatise

L’éditeur web se connecte à votre repository Git via notre [GitHub App](/fr/settings/github) ou notre [intégration GitLab](/fr/settings/gitlab) et gère Git automatiquement. Lorsque vous :

- **Ouvrez un fichier** : récupère la dernière version depuis votre repository.
- **Modifiez un fichier** : suit vos modifications comme un brouillon qui peut devenir un commit.
- **Enregistrez les modifications** : crée un commit avec vos modifications.
- **Créez une branche** : crée une nouvelle branche dans votre repository.
- **Publiez sur votre branche de déploiement** : crée un commit et pousse directement sur votre branche de déploiement.
- **Publiez sur d’autres branches** : crée un commit et ouvre une pull request.

### Quand utiliser les branches

Les branches vous permettent de travailler sur des modifications sans affecter le contenu de votre site en ligne. Lorsque vos changements sont prêts, vous pouvez les fusionner dans votre branche de déploiement avec une pull request.

**Bonne pratique : travaillez toujours à partir de branches.** Cela garantit que votre documentation en production reste stable et permet des workflows de revue appropriés.

## Créer une branche

1. Sélectionnez le nom de la branche dans la barre d’outils de l’éditeur (généralement `main` par défaut).
2. Sélectionnez **New Branch**.
3. Saisissez un nom descriptif pour votre branche, par exemple `update-getting-started` ou `fix-installation-steps`.
4. Sélectionnez **Create Branch**.

<Note>
  Vous devrez peut-être sélectionner votre nouvelle branche dans le menu déroulant si l’éditeur ne bascule pas automatiquement dessus.
</Note>

### Enregistrer des modifications sur une branche

Pour enregistrer vos modifications sur une branche, cliquez sur le bouton **Save Changes** dans l’angle supérieur droit de l’éditeur. Cela crée un commit avec vos modifications et le pousse sur votre branche.

### Changer de branche

1. Sélectionnez le nom de la branche actuelle dans la barre d’outils de l’éditeur.
2. Choisissez la branche vers laquelle vous souhaitez basculer dans le menu déroulant.

<Tip>
  Toute modification non enregistrée sera perdue lors du changement de branche. Assurez-vous d’enregistrer ou de publier votre travail avant de basculer.
</Tip>

## Apporter des modifications

Modifiez le contenu existant, créez de nouvelles pages et organisez la structure de votre site dans l’Éditeur web.

### Navigation dans les fichiers

Utilisez l’explorateur de fichiers de la barre latérale pour parcourir votre documentation, ou appuyez sur <kbd>Command</kbd> + <kbd>P</kbd> (<kbd>Ctrl</kbd> + <kbd>P</kbd> sous Windows) pour rechercher des fichiers.

### Modification de contenu

Modifiez vos pages en mode visuel ou en mode Markdown.

En mode visuel, appuyez sur <kbd>/</kbd> pour ouvrir le menu déroulant des composants. Ajoutez des blocs de contenu, des encarts, des blocs de code et d’autres composants pour personnaliser votre documentation.

### Mise à jour de la navigation

Ajoutez, supprimez et réorganisez des pages dans votre navigation en modifiant votre fichier `docs.json` :

1. Accédez à votre `docs.json` dans l’explorateur de fichiers.
1. Mettez à jour la propriété `navigation` pour obtenir la structure de navigation souhaitée. Voir [Navigation](/fr/navigation) pour plus d’informations.

Cet exemple montre comment ajouter une page Themes au groupe Profile.

```json title="Adding a Themes page" highlight="18"
{
    "navigation": {
        "groups": [
            {
                "group": "Getting started",
                "pages": [
                    "index",
                    "quickstart",
                    "installation"
                ]
            },
            {
                "group": "Profile",
                "pages": [
                    "settings",
                    "account-types",
                    "dashboard",
                    "themes"
                ]
            }
        ]
    }
}
```

## Publication des modifications

Cliquez sur le bouton **Publish** pour enregistrer vos modifications et les rendre disponibles.

La façon dont vos modifications sont publiées dépend de la branche sur laquelle vous travaillez :

* **Branche de déploiement** : met immédiatement à jour votre site en production.
* **Autres branches** : crée une pull request pour vous permettre de vérifier les modifications avant leur mise en ligne.

## Pull requests et revue des modifications

Les pull requests vous permettent de proposer des modifications depuis votre branche afin que d'autres personnes puissent les examiner avant de les intégrer à votre documentation en production. Cela aide à garantir l'exactitude de vos modifications et offre à votre équipe l'occasion de collaborer sur le contenu.

<Tip>
  Même si vous travaillez seul, les pull requests sont utiles pour prévisualiser les modifications avant leur mise en ligne et pour maintenir un historique clair des mises à jour.
</Tip>

### Créer une pull request à partir d’une branche 

Lorsque vous êtes prêt à publier les modifications de votre branche :

1. Assurez-vous que toutes vos modifications sont enregistrées sur votre branche.
1. Sélectionnez **Publish Pull Request** dans le coin supérieur droit de l’éditeur.
1. Ajoutez un titre et une description à votre pull request. Un bon titre et une bonne description aident les relecteurs à comprendre les modifications que vous avez effectuées.
1. Sélectionnez **Publish Pull Request**.

L’éditeur web créera une pull request dans votre dépôt Git et fournira un lien pour la consulter.

### Revoir les pull requests

Une fois votre pull request créée :

1. **Passez en revue vos modifications** : Vous et les membres de votre équipe pouvez examiner votre pull request dans votre fournisseur Git, comme GitHub ou GitLab.
1. **Apportez des modifications supplémentaires** : Après la revue, effectuez les modifications nécessaires dans votre éditeur web. L’enregistrement de vos changements mettra à jour votre pull request.
1. **Fusionnez quand vous êtes prêt** : Lorsque votre pull request est prête, fusionnez-la pour déployer les modifications sur votre site de documentation en production.

## Dépannage

Voici des solutions aux problèmes courants que vous pouvez rencontrer avec l’Éditeur web.

<AccordionGroup>
  <Accordion title="Les modifications n’apparaissent pas après la publication">
    **Causes possibles :**

    - Le déploiement est toujours en cours
    - Problèmes de cache dans votre navigateur

    **Solutions :**

    1. Vérifiez l’état du déploiement dans votre tableau de bord Mintlify.
    2. Essayez d’actualiser la page de force (<kbd>Ctrl</kbd> + <kbd>F5</kbd> ou <kbd>Cmd</kbd> + <kbd>Shift</kbd> + <kbd>R</kbd>).
    3. Videz le cache de votre navigateur.
  </Accordion>
  <Accordion title="Erreurs d’autorisation lors de la publication">
    **Causes possibles :**

    - Autorisations insuffisantes sur le dépôt Git
    - Problèmes d’authentification avec votre fournisseur Git

    **Solutions :**

    1. Vérifiez que vous disposez des droits d’accès requis au dépôt.
    2. Assurez-vous que votre intégration Git est correctement configurée.
    3. Consultez la documentation [Permissions de l’éditeur](/fr/advanced/dashboard/permissions).
  </Accordion>
  <Accordion title="Problèmes de chargement de l’éditeur">
    **Causes possibles :**

    - Problèmes de connectivité réseau
    - Dépôts de documentation volumineux

    **Solutions :**

    1. Vérifiez votre connexion Internet.
    2. Actualisez la page et réessayez.
    3. Contactez le support si le problème persiste.
  </Accordion>
</AccordionGroup>
