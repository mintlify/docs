L'éditeur web enregistre automatiquement tout ce que vous tapez, mais vos modifications ne sont mises en ligne que lorsque vous choisissez de les publier.

Ce qui se passe lorsque vous publiez dépend de deux choses : **la branche sur laquelle vous vous trouvez** et **si cette branche nécessite des pull requests**.

<div id="saving-versus-publishing">
  ## Enregistrement vs publication
</div>

**L'enregistrement** se fait automatiquement. Vos modifications sont stockées sur les serveurs de Mintlify et persistent à travers les onglets du navigateur, les appareils et les interruptions du réseau.

**La publication** valide vos modifications dans votre dépôt Git. Cliquez sur le bouton **Publier** dans la barre d'outils pour ouvrir le menu de publication, enregistrer vos modifications sous forme de commit Git et créer une pull request.

L'éditeur suit les éléments suivants comme modifications en attente :

- Modifications de contenu dans les pages
- Pages nouvelles ou supprimées
- Modifications de la structure de navigation
- Téléversements de médias
- Mises à jour de configuration

<div id="what-happens-when-you-publish">
  ## Que se passe-t-il lorsque vous publiez
</div>

Les actions disponibles lorsque vous cliquez sur le bouton publier dépendent de votre branche actuelle et de l'éventuelle existence de règles de protection de branche qui exigent des pull requests.

| Type de branche | Protection de branche | Actions disponibles |
|-------------|-------------------|-------------------|
| <Tooltip headline="Branche de déploiement" tip="La branche qui se publie sur votre site de documentation en direct, généralement 'main'.">Branche de déploiement</Tooltip> | Aucune | **Publier** directement sur votre site en direct |
| Branche de déploiement | Pull requests requises | **Créer une branche** pour déplacer les modifications vers une nouvelle branche |
| <Tooltip headline="Branche de fonctionnalité" tip="Une branche isolée où vous travaillez sur les mises à jour avant de fusionner vers votre branche de déploiement.">Branche de fonctionnalité</Tooltip> | Aucune | **Enregistrer dans la branche**, **Créer une pull request** |
| Branche de fonctionnalité | Pull requests requises | **Enregistrer dans la branche**, **Créer une pull request** |

- **Publier** : Valide et déploie vos modifications sur votre site en direct immédiatement.
- **Enregistrer dans la branche** : Valide vos modifications sur la branche de fonctionnalité sans fusionner vers votre branche de déploiement.
- **Créer une branche** : Déplace vos modifications en attente vers une nouvelle branche de fonctionnalité lorsque vous êtes sur une branche de déploiement protégée.
- **Créer une pull request** : Ouvre une pull request ciblant votre branche de déploiement.

S'il n'y a pas de modifications en attente, les actions de publication et d'enregistrement sont désactivées.

<Note>
  Votre site en direct est mis à jour après que Mintlify a construit et déployé vos modifications. Cela prend généralement de 30 secondes à quelques minutes. Vérifiez l'état du déploiement sur votre [tableau de bord](https://dashboard.mintlify.com).
</Note>

<div id="when-to-use-a-branch">
  ## Quand utiliser une branche
</div>

**Modifiez directement sur votre branche de déploiement** si vous n'utilisez pas de workflow basé sur Git.

**Créez une branche** lorsque vous utilisez un workflow docs-as-code où chaque modification de votre contenu est effectuée sur une branche distincte.

<div id="create-and-switch-branches">
  ## Créer et changer de branche
</div>

<div id="create-a-branch">
  ### Créer une branche
</div>

1. Cliquez sur le nom de la branche dans la barre d'outils de l'éditeur.
1. Cliquez sur **Créer une nouvelle branche**.
1. Si vous avez des modifications en attente, choisissez de les apporter à la nouvelle branche ou de les laisser sur la branche actuelle.
1. Saisissez un nom et cliquez sur **Créer une branche**.

<Tip>
  Utilisez des noms de branche descriptifs afin de pouvoir les identifier facilement et que les autres comprennent à quoi sert chaque branche.
</Tip>

<div id="switch-branches">
  ### Changer de branche
</div>

1. Cliquez sur le nom de la branche dans la barre d'outils.
1. Recherchez ou faites défiler jusqu'à la branche souhaitée.
1. Cliquez sur la branche pour y basculer.

<Note>
  Le changement de branche alors que vous avez des modifications non publiées vous demandera d'apporter ces modifications à la nouvelle branche ou de les laisser. Les modifications laissées restent sur votre branche d'origine.
</Note>

<Tip>
  Pour copier le nom d'une branche, survolez la branche dans la liste déroulante et cliquez sur l'icône de copie. Ceci est utile lorsque vous partagez la branche avec des coéquipiers ou que vous y faites référence dans une pull request.
</Tip>

<div id="preview-your-changes">
  ## Prévisualiser vos modifications
</div>

Chaque fois que vous enregistrez des modifications sur une branche de fonctionnalité, Mintlify crée un déploiement d'aperçu — une URL temporaire où vos modifications sont rendues exactement telles qu'elles apparaîtront une fois publiées.

<div id="access-and-share-a-preview">
  ### Accéder à un aperçu et le partager
</div>

1. Cliquez sur **Publier** dans la barre d'outils de l'éditeur.
1. Dans le menu de publication, cliquez sur l'URL de l'aperçu. Le format de l'URL est `organization-branch-name.mintlify.app`.
    <Frame>
      <img src="/images/editor/preview-url-light.png" alt="URL d'aperçu mise en évidence dans le menu de publication." className="block dark:hidden" />
      <img src="/images/editor/preview-url-dark.png" alt="URL d'aperçu mise en évidence dans le menu de publication." className="hidden dark:block" />
    </Frame>

Copiez l'URL et envoyez-la aux relecteurs. L'aperçu est mis à jour automatiquement chaque fois que vous enregistrez sur la branche.

<div id="restrict-access-to-previews">
  ### Restreindre l'accès aux aperçus
</div>

Les URL d'aperçu sont accessibles publiquement par défaut. Pour restreindre l'accès aux membres de votre organisation Mintlify, activez l'authentification d'aperçu sur la page [Add-ons](https://dashboard.mintlify.com/products/addons) de votre tableau de bord.

<div id="share-editor-links">
  ### Partager les liens de l'éditeur
</div>

Pour inviter un coéquipier à une page spécifique sur une branche, copiez l'URL depuis la barre d'adresse de votre navigateur et partagez-la. Toute personne ayant accès à votre organisation Mintlify peut ouvrir le lien directement dans sa session d'éditeur.

Le format de l'URL est :

```text
https://dashboard.mintlify.com/{org}/{project}/editor/{branch}/~/{filepath}
```

Par exemple : `https://dashboard.mintlify.com/acme/docs/editor/main/~/guides/quickstart.mdx`

<div id="review-and-merge-pull-requests">
  ## Examiner et fusionner les pull requests
</div>

Lorsqu'une pull request est ouverte pour la branche actuelle, le menu de publication affiche son statut de révision :

- **Approuvée** : La pull request a été approuvée et est prête à être fusionnée.
- **Modifications demandées** : Un relecteur a demandé des modifications avant que la pull request puisse être fusionnée.
- **En attente de révision** : La pull request attend un relecteur.

Cliquez sur **Voir la requête** pour ouvrir la pull request dans votre fournisseur Git.

Lorsque la pull request est approuvée, cliquez sur **Fusionner et publier** pour fusionner et déployer directement depuis l'éditeur. L'éditeur passe à votre branche de déploiement après la fusion.

<Tip>
  Configurez les règles de protection de branche dans votre fournisseur Git pour exiger des pull requests. Consultez [À propos des branches protégées](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches) dans la documentation GitHub ou [Branches protégées](https://docs.gitlab.com/user/project/repository/branches/protected/) dans la documentation GitLab.
</Tip>

<div id="review-changes-before-merging">
  ### Examiner les modifications avant la fusion
</div>

Cliquez sur n'importe quel fichier modifié dans le menu de publication pour l'ouvrir en mode diff et comparer votre branche à la version publiée. Les fichiers qui ne peuvent pas afficher de diff, tels que les images ou les fichiers supprimés, apparaissent dans la liste mais ne sont pas cliquables.

<div id="simultaneous-publishing">
  ### Publication simultanée
</div>

Une seule publication peut avoir lieu à la fois par branche. Si un autre membre de l'équipe publie sur la même branche, attendez la fin de la publication en cours avant de réessayer.

<div id="commit-messages">
  ### Messages de commit
</div>

Lorsque vous publiez, vous pouvez saisir un message de commit avant de confirmer. Si vous le laissez vide, l'éditeur génère un message basé sur les fichiers modifiés.

<div id="resolve-conflicts">
  ## Résoudre les conflits
</div>

Les conflits surviennent lorsque votre branche et la branche de déploiement ont des modifications incompatibles sur les mêmes fichiers. Par exemple, lorsque vous et un coéquipier avez modifié les mêmes lignes dans un fichier ou déplacé un fichier vers des emplacements différents.

L'éditeur affiche un avertissement lorsque des conflits bloquent la publication. Suivez les invites pour choisir quelle version de chaque section conflictuelle conserver.

<div id="collaborate-in-real-time">
  ## Collaborer en temps réel
</div>

Lorsque plusieurs personnes ouvrent la même page sur la même branche, elles modifient ensemble en temps réel. Le curseur et les modifications de chacun sont visibles pour tous, avec les avatars affichés dans la barre d'outils.

- Les modifications de tous les collaborateurs fusionnent automatiquement. Deux personnes modifiant la même section ne créeront pas de conflits.
- L'annulation n'affecte que vos propres modifications.
- Si vous perdez votre connexion, les modifications sont enregistrées localement et synchronisées lors de la reconnexion.

Lorsque l'[agent Mintlify](/fr/agent) modifie une page via l'API ou le MCP, il apparaît dans l'éditeur comme tout autre collaborateur. Vous voyez l'avatar de l'agent dans la barre d'outils et un curseur en direct à son emplacement de modification le plus récent. Le curseur s'efface automatiquement une fois l'agent terminé.

<div id="git-sync">
  ## Synchronisation Git
</div>

Lorsque quelqu'un pousse des modifications sur votre dépôt en dehors de l'éditeur, l'éditeur intègre ces modifications automatiquement.

Les modifications non chevauchantes s'appliquent automatiquement. Si une modification distante et votre modification locale affectent la même partie d'une page, l'éditeur met en évidence le conflit afin que vous puissiez le résoudre.

<div id="commit-signing">
  ## Signature des commits
</div>

Signez les commits avec votre compte GitHub en l'autorisant dans les [paramètres de votre compte](https://dashboard.mintlify.com/settings/account). Sans autorisation, l'application GitHub Mintlify signe les commits effectués dans l'éditeur web.

<div id="git-operations-reference">
  ## Référence des opérations Git
</div>

À titre de référence, voici comment les actions de l'éditeur correspondent aux opérations Git sous-jacentes.

| Action dans l'éditeur | Opération Git |
|---|---|
| Modifier une page | Les modifications sont enregistrées automatiquement sur les serveurs Mintlify. Pas encore de commit Git. |
| Publier sur votre branche de déploiement | `git commit` et `git push`. Déclenche un déploiement. |
| Enregistrer dans la branche | `git commit` sur la branche de fonctionnalité actuelle. |
| Créer une pull request | `git push` et ouvre une pull request contre votre branche de déploiement. |
| Fusionner et publier | Fusionne la pull request et déclenche un déploiement. |
| Créer une branche | `git checkout -b <branch-name>` |
| Changer de branche | `git checkout <branch-name>` |
| Push externe ou mise à jour CLI | Les modifications entrantes se synchronisent automatiquement dans l'éditeur via une fusion à trois voies. |
